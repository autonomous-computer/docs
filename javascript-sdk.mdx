Use the JavaScript SDK in a TypeScript or JavaScript service when you want a client wrapper around the REST contract. Keep the API key in a trusted server-side environment, not in browser code.

## Install and configure

```bash
npm install @secapi/sdk-js
export SECAPI_API_KEY="secapi_..."
```

```ts
import { SecApiClient } from "@secapi/sdk-js"

const client = new SecApiClient({
  apiKey: process.env.SECAPI_API_KEY!,
})
```

Do not expose `SECAPI_API_KEY` to browser code. Call the SDK from a server route, worker, background job, or other trusted runtime.

## Make a first request

```ts
const issuer = await client.resolveEntity({ ticker: "AAPL" })
const filing = await client.agentLatestFiling({
  ticker: "AAPL",
  form: "10-K",
})

console.log({
  ticker: issuer.ticker,
  cik: issuer.cik,
  accessionNumber: filing.accessionNumber,
  filingUrl: filing.filingUrl,
  requestId: filing.requestId,
})
```

`resolveEntity` calls `GET /v1/entities/resolve`. `agentLatestFiling` calls the latest-filings endpoint with `view=agent`. The returned identifier and filing fields vary with the current filing, so use this output to confirm the request and preserve it with downstream work. Use `latestFiling` when you need the endpoint's default response instead.

For accepted parameters, response fields, and endpoint limits, see the [entity resolution reference](/api-reference/entities/get-v1-entities-resolve) and [latest filing reference](/api-reference/filings/get-v1-filings-latest).

## Retrieve a filing section

```ts
const riskFactors = await client.agentSection({
  ticker: "AAPL",
  form: "10-K",
  sectionKey: "item_1a",
})
```

`agentSection` requests the latest matching section with `view=agent`. Preserve the returned filing identifiers and request ID with an extracted section or a conclusion built from it.

## Handle errors

```ts
import { SecApiClient, SecApiError } from "@secapi/sdk-js"

try {
  await client.resolveEntity({ ticker: "AAPL" })
} catch (error) {
  if (error instanceof SecApiError) {
    console.error({
      status: error.status,
      code: error.code,
      requestId: error.requestId,
      message: error.message,
    })
  }
}
```

The SDK handles its documented retry behavior for safe reads. For `429`, let the client honor the service's retry timing; do not add concurrent retry loops. Only opt into retries for a mutation when your operation is idempotent. Read [SDK reliability](/sdk-reliability) and [Troubleshooting](/troubleshooting) before adding application-level retries.
