---
title: "Evaluate SEC Filings MCP Server"
description: "Verify one authenticated MCP filing workflow and the source evidence it returns before enabling an agent."
---

Use hosted MCP when an agent client needs to discover and call SEC API tools. This is a transport and evidence evaluation, not a certification that an agent’s summary is correct. Use REST or an SDK when your server owns the workflow directly or requires the full documented HTTP surface.

## Reader job: prove an agent can return a filing source

Connect one supported MCP client with an environment-backed API key. Discover the deployed tool inventory, resolve an issuer, retrieve a filing and section, then confirm the same source with REST. The result should be a reproducible filing record, not just fluent prose.

## Configure and verify

Public discovery describes the endpoint but does not authenticate a client or grant tool access:

```bash
export SECAPI_API_KEY="secapi_..."
curl --fail-with-body -sS https://api.secapi.ai/mcp
```

Configure the client to send `x-api-key` from its secret environment. Ask it to discover tools and use `entities.resolve`, `filings.latest` or `filings.search`, and `sections.get` for `AAPL` and `Item 1A`. The deployed inventory is the compatibility contract for that session.

Cross-check the source record over REST:

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"

curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&mode=full"
```

`GET /mcp` is public metadata. Authenticated JSON-RPC tool calls use `POST /mcp`; do not present discovery as anonymous MCP access.

## What a passing test retains

Keep a redacted client configuration record, client name, discovered tool names, arguments, accession number, filing URL, section key, returned text, `requestId`, `traceparent`, and any provenance or freshness fields. REST and MCP should identify the same source record. They do not need to produce identical agent wording.

## Operational boundaries

- MCP exposes the tools advertised by the deployed session, not an open-ended promise of REST parity. Use REST or an SDK for capabilities absent from discovery, such as a filing-diff report.
- Keep the API key in the client or server secret facility; do not place it in an agent prompt, browser bundle, or transcript.
- A response without filing identity fails the evidence test, even when the narrative is plausible.
- Distinguish authentication, authorization, validation, quota or billing, and tool-execution failures. Preserve identifiers before retrying a bounded call.
- Filing text is source material, not legal, accounting, or investment advice. An agent still needs an answer policy and human-review path where your use case requires one.

## Decision point

Enable an MCP workflow only when the chosen client can retain source identifiers and surface failed retrieval. Otherwise, keep the credential and retrieval loop behind your own server, then pass verified source text to the agent.
