Evaluate MCP the same way you evaluate REST: verify discovery, authenticate from the environment, and require a filing citation from one complete workflow.

## Test one evidence-preserving workflow

<Steps>
  <Step title="Read live discovery metadata">
    Inspect `https://api.secapi.ai/mcp` before configuring a client.
  </Step>
  <Step title="Keep the credential out of source">
    Configure the hosted streamable HTTP server with an environment-backed `x-api-key`, not a checked-in secret.
  </Step>
  <Step title="Complete one cited retrieval">
    Resolve an issuer, fetch its latest 10-K, extract Item 1A, and require the accession number, SEC filing URL, and request ID alongside the text.
  </Step>
</Steps>

The equivalent REST calls make the expected result concrete:

```bash
curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"

curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K"
```

## What to verify

- The client does not expose a key in source control or a browser-visible URL.
- Tool output identifies the source filing and preserves failure details.
- The consuming application keeps `requestId`, `traceparent`, and source metadata when returned.

<Prompt>
After connecting to `https://api.secapi.ai/mcp` with an environment-backed `x-api-key`, use `tools.search` to find issuer resolution, latest filing, and filing-section capabilities. Use `tools.describe` before supplying arguments. For AAPL, retrieve the latest 10-K Item 1A and return the accession number, filing URL, section title, request ID, `traceparent` when present, and any truncation or freshness signal.
</Prompt>

## Common failures

- Old installation notes: read deployed discovery metadata again.
- Missing or invalid API key: pass `x-api-key` from the environment.
- Uncited text: retrieve the filing or section again before use downstream.
