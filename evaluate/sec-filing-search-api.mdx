Use this page to assess a filing-search layer for an application or research workflow. Choose it when you need a bounded, filterable filing manifest plus content retrieval, rather than asking users to navigate raw EDGAR or relying on a general search result. The concrete job is to locate a filing for one issuer, search it for a question, and retrieve the section the user will inspect. A result list is not a substitute for filing text.

## Endpoints and boundaries

- `GET /v1/filings` returns a paginated filing-manifest list. It supports historical filters, sorting, and cursor pagination.
- `GET /v1/filings/latest` returns the latest filing matching an issuer and optional form.
- `GET /v1/search/fulltext` finds text matches; `GET /v1/search/semantic` finds section matches for semantic retrieval.
- `GET /v1/filings/{accession_number}/sections/{section_key}` retrieves the source section.

Scope issuer identity with a ticker, symbol, or CIK and avoid conflicting identifiers. `limit` is 1 through 50 and the filing cursor is a zero-based offset; do not reuse a cursor after changing filters. A missing result is a retrieval outcome, not proof an issuer did not disclose something.

## Selection test

Use filing manifests when the application needs an accession-number handoff to retrieval or extraction. Use full-text search for literal wording and semantic search for a concept expressed differently in the filing; neither ranking is a completeness claim. The proof passes when a user can reproduce the selected result from its filters, open the resulting section, and distinguish an empty or unavailable search path from a disclosure conclusion.

## Proof of concept

```bash
# 1. Identify a recent annual report.
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AAPL&form=10-K&sort=filing_date_desc&limit=1"

# 2. Search filing content for a review question.
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/fulltext?q=material%20weakness&ticker=AAPL&form=10-K&limit=10"

# 3. Fetch the selected section with its actual accession number and section key.
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/ACCESSION/sections/SECTION_KEY?ticker=AAPL&form=10-K"
```

Replace the placeholders with result values. Test an issuer your workflow actually covers, then paginate the list if the result will feed an ingestion process.

## Expected evidence

The evaluation artifact should preserve accession number, filing URL, form, filing date, issuer identity, source section key, `requestId`, and returned provenance/freshness fields. A reviewer should be able to open the cited filing and reproduce why the application selected that section.

## Failure modes

- A bad filter, invalid limit, or conflicting issuer identifier is a request-validation failure to correct.
- A search result without a fetched source section is retrieval assistance, not a quote.
- Pagination changes after filters change; restart rather than mixing cursor sequences.
- Full-text and semantic search have different dependencies and may fail independently. Handle an unavailable backend distinctly from an empty result set.
