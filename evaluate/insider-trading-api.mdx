Use this page to evaluate the insider endpoint before you publish alerts or feed its results into an investor workflow. A successful evaluation produces reviewable Form 3, 4, or 5 records with their filing evidence; it does not validate an investment signal or a claim about insider intent.

## Run a narrow Form 4 query

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/insiders?ticker=AAPL&forms=4&limit=10"
```

Then test a transaction-code filter:

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/insiders?ticker=JPM&forms=4&transaction_code=P&limit=25"
```

## Verify your implementation

<Steps>
  <Step title="Narrow one query">
    Check that issuer, reporting owner, form, filing date, and transaction date are represented separately.
  </Step>
  <Step title="Preserve filing fields together">
    Keep `transactionCode` and `transactionDirection` with accession number, `provenance.filingUrl`, and `requestId` in every alert.
  </Step>
  <Step title="Exercise pagination">
    Follow `nextCursor` only when `hasMore` is true.
  </Step>
  <Step title="Keep sparse records sparse">
    Test absent price, security-title, or ownership values without replacing filing-derived gaps with estimates.
  </Step>
</Steps>

## Evaluation prompt

<Prompt>
Use SEC API to retrieve recent Form 4 records for one issuer. Return reporting owner, title when present, transaction and filing dates, transaction code, direction, shares and price when present, accession number, filing URL, and request ID. Do not infer motive, open-market activity, or an investment recommendation from a code without reviewing the cited filing and footnotes.
</Prompt>

## Interpretation limits

A Form 4 can cover grants, exercises, tax withholding, gifts, indirect ownership, and planned activity. A transaction code is not a complete explanation of the event. A non-2xx response is a failed lookup, not evidence of no insider activity; retain request metadata for diagnosis.
