## What to test

- Entity resolution before retrieval
- Latest filing and historical filing search
- Section-level context for Item 1A, Item 7, and other analyst workflows
- Semantic search for open-ended research questions
- Compact output mode and token/cost headers
- Citation discipline: accession number, filing date, source URL, and request metadata

## Proof-of-concept prompt

<Prompt>
Evaluate SEC API as the retrieval layer for SEC filing RAG. For TSLA, resolve the entity, retrieve the latest 10-K, extract Item 1A and Item 7, run semantic search for supply-chain and regulatory risk, and produce a cited answer that separates retrieved filing text from interpretation. Keep accession number, filing URL, requestId, freshness, and token/cost headers visible.
</Prompt>

## Evaluation path

<Steps>
  <Step title="Keep retrieval scoped">
    Pull sections, facts, and snippets before raw full filings. The best RAG loop is smaller, cited, and easier to verify.
  </Step>
  <Step title="Force citation structure">
    Require every claim to carry filing identity and request metadata. Do not accept ungrounded prose as a successful RAG test.
  </Step>
  <Step title="Measure context cost">
    Compare compact and full retrieval shapes before picking a default for agents.
  </Step>
</Steps>
