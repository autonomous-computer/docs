---
title: "Evaluate SEC Filing RAG"
description: "Test whether a filing-answer workflow can retrieve and retain the SEC source behind each answer."
---

Use this guide when you are evaluating a filing Q&A product, research assistant, or internal agent. SEC API can retrieve candidate sections and source identity. Your application is responsible for answer generation, citation display, review rules, and deciding whether the source supports the answer.

## Test one answer a reviewer can check

Pick one issuer, one question, and one form. Search for candidates, fetch the selected section, and compare a literal query with a conceptual one. The test passes when a reviewer can locate the filing and read the relevant disclosure, not when an answer merely sounds plausible.

## Retrieve candidates, then verify the source

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/semantic?q=regulatory%20risk&ticker=TSLA&form=10-K&mode=hybrid&limit=5"

curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/fulltext?q=regulatory%20risk&ticker=TSLA&form=10-K&limit=5"

curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/ACCESSION/sections/SECTION_KEY?ticker=TSLA&form=10-K&mode=full"
```

Use `GET /v1/search/semantic` when the question may not use the filing's wording. Use `GET /v1/search/fulltext` for literal language. Search results are leads, not the text to quote. Fetch the section using the returned accession and section key, and open the SEC filing URL where tables or context matter.

## What to store with an answer

Keep the query, filters, accession number, section key, form, filing date, filing URL, `requestId`, and returned provenance or freshness fields. Preserve `char_start`, `char_end`, and `_citation_degraded` when semantic search provides them. Those offsets refer to normalized section markdown, not raw SEC HTML.

## When it fits, and when it does not

This workflow fits an answer product that can attach a fetched source section to material assertions and show failed retrieval to a reviewer. It is not the right fit for a workflow that needs an unsupported guarantee of corpus completeness, a materiality decision, or a substitute for legal, accounting, or investment judgment. Use a filing-first human review when the conclusion depends on tables, definitions, or document layout.

## Make failures visible

- An empty result means this request found no match; it does not establish that an issuer made no disclosure.
- A `503` or degraded-state signal can indicate unavailable or partial search. Return that state rather than manufacture a negative answer.
- A snippet is navigation evidence. Fetch the section before quoting it.
- Store the resolved accession whenever a workflow uses a latest route, because the result can change between runs.

## Next step

Expand from one question only after your interface can show the source record and the retrieval state beside the answer. Price the bounded search-and-section sequence before widening the issuer universe or refresh cadence.
