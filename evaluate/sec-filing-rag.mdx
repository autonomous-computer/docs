Use this page when you are evaluating a filing Q&A, research assistant, or internal agent. SEC API can find candidate filing sections and return source identity. Your application remains responsible for answer generation, citation display, review policy, and deciding whether the retrieved evidence supports an answer.

## Reader job: prove one answer can be checked

Choose one issuer, one question, and one 10-K. Retrieve candidate passages, fetch the selected section by accession number, and compare the exact-language and conceptual search paths. A good proof does not ask whether the model sounds convincing; it asks whether a reviewer can find the same filing and read the surrounding disclosure.

## Run the evidence path

Use semantic search for a question whose wording may differ from the filing. Use full-text search separately for language you expect to appear literally. Neither result is the source text you should quote.

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/semantic?q=regulatory%20risk&ticker=TSLA&form=10-K&mode=hybrid&limit=5"

curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/fulltext?q=regulatory%20risk&ticker=TSLA&form=10-K&limit=5"

curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/ACCESSION/sections/SECTION_KEY?ticker=TSLA&form=10-K&mode=full"
```

Replace `ACCESSION` and `SECTION_KEY` with values returned by search. The last request is the verification step: it returns the named extracted section and filing provenance. Open the returned SEC filing URL when layout, tables, definitions, or surrounding context matter.

## What SEC API supplies, and what it does not

| SEC API capability                                          | Use it for                                                              | Do not treat it as                                            |
| ----------------------------------------------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------------- |
| `GET /v1/search/semantic`                                   | Conceptually related normalized sections, with citation-oriented fields | Corpus-wide coverage or a materiality ranking                 |
| `GET /v1/search/fulltext`                                   | Literal terms across filing and section search legs                     | A single globally ranked, paginated evidence list             |
| `GET /v1/filings/{accession_number}/sections/{section_key}` | The selected section text and filing identity                           | A guarantee of coverage for tables, exhibits, or topics       |

Retain the query and filters, accession number, section key, form, filing date, source URL, `requestId`, and returned `provenance` or `freshness` fields. Preserve `char_start`, `char_end`, and `_citation_degraded` when semantic search supplies them. Those offsets refer to normalized section markdown, not raw SEC HTML.

## Evaluation failure cases

- An empty result means this request found no match. It does not establish that the issuer made no disclosure.
- A `503` or a returned degraded-state signal means a search dependency was unavailable or partial. Show failed retrieval rather than generating a negative answer.
- A snippet is a navigation aid. Fetch the section before quoting it, and use the SEC source to inspect tables and qualifications.
- The latest filing can change between runs. Store the resolved accession whenever a workflow intentionally uses a latest route.

## Decide whether to expand

Move from a one-question test to an agent loop only when your product can attach a fetched source section to the filing assertions it presents as material and can expose retrieval failure to a reviewer. Price the bounded request sequence before widening issuers, forms, or cadence.
