---
title: "SEC API status"
description: "Check service availability and collect the evidence needed to diagnose an API or MCP request"
---

Start with the public status page and a health request. A healthy service check does not prove that a particular issuer, filing, derived dataset, or upstream SEC source is available.

```bash
curl -sS "https://api.secapi.ai/healthz"
```

## Public endpoints

- Status: [status.secapi.ai](https://status.secapi.ai)
- API: `https://api.secapi.ai`
- Hosted MCP: `https://api.secapi.ai/mcp`
- Health: `https://api.secapi.ai/healthz`
- Readiness: `https://api.secapi.ai/readyz`

`/healthz` is a service-health check. `/readyz` is a readiness check. Use the response from the endpoint you call as the evidence for that check; neither replaces route-specific source, provenance, freshness, or degradation fields.

## When a request fails

1. Check [status.secapi.ai](https://status.secapi.ai) and request `/healthz` once.
2. Reproduce the API or MCP request once with the same endpoint and safe, redacted inputs.
3. For rate limits, honor `Retry-After`; do not loop on quota, billing-cap, or authorization failures.
4. Keep the endpoint, timestamp, HTTP status, error `code`, `Request-Id`, and `traceparent` when present.
5. Include route-specific source and state fields for a data-result issue, but never send an API key or private payload.

Read [Troubleshooting](/troubleshooting) for status and error-code guidance, and [Freshness and source evidence](/freshness-and-trust) for the distinction between a service check and a data result.
