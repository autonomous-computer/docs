title: "SEC API status"
description: "Check public service availability and retain diagnostic evidence for an API or MCP request"
---

Use the public status page and a health request to assess service availability. Neither proves that a specific issuer, filing, route, derived dataset, or upstream SEC source is available for the request you need.

```bash
curl -sS "https://api.secapi.ai/healthz"
```

## Public service checks

| Surface | Address |
| --- | --- |
| Status | [status.secapi.ai](https://status.secapi.ai) |
| API | `https://api.secapi.ai` |
| Hosted MCP | `https://api.secapi.ai/mcp` |
| Health | `https://api.secapi.ai/healthz` |
| Readiness | `https://api.secapi.ai/readyz` |

`/healthz` reports service health; `/readyz` reports readiness. Preserve the response as evidence for the check you actually called. Route-level provenance, freshness, and degradation fields remain the evidence for a data result.

## Diagnose one request

1. Review [status.secapi.ai](https://status.secapi.ai), then make one health request.
2. Reproduce the API or MCP request once with the same endpoint and redacted inputs.
3. Honor `Retry-After` for rate limits; do not loop on quota, billing-cap, or authorization failures.
4. Retain endpoint, UTC timestamp, HTTP status, error `code`, `Request-Id`, and `traceparent` when supplied.
5. For a data issue, retain the returned source and state fields. Never send an API key or private payload in a support report.

Use [Request diagnostics](/request-diagnostics) for a retained request ID, [Troubleshooting](/troubleshooting) for error handling, and [Freshness and source evidence](/freshness-and-trust) to distinguish a healthy service from evidence-ready data.
