Use the public status page for incidents and the health endpoints for a quick service check. A healthy endpoint does not prove that a specific issuer, derived dataset, or upstream SEC source is available; it only describes the service check that ran.

## Public endpoints

- Status: [status.secapi.ai](https://status.secapi.ai)
- API: `https://api.secapi.ai`
- Hosted MCP: `https://api.secapi.ai/mcp`
- Health: `https://api.secapi.ai/healthz`
- Readiness: `https://api.secapi.ai/readyz`

## What to capture

For a customer request handled by the SEC API app, retain the response `Request-Id` and `traceparent`, the endpoint, timestamp, HTTP status, and error code. A client may supply a correlation ID for tracing, but correlation IDs are not billing idempotency keys.

For a data result, also retain the route-specific filing accession, source URL, provenance, freshness, or degradation fields that are present. See [Freshness, Coverage, and Source Evidence](/freshness-and-trust) for why these are separate from a service-health check.

## When something fails

1. Check [status.secapi.ai](https://status.secapi.ai) and `GET /healthz`.
2. Reproduce the request once with the same endpoint and safe, redacted inputs.
3. Do not repeat quota, billing-cap, or rate-limit requests in a loop; honor `Retry-After` when present.
4. Open a support request with the request evidence, not an API key or private payload.
