---
title: "Build a Filing Monitor"
description: "Save a filing search, deliver signed monitor matches, and reconcile result history."
---

Use a monitor for scheduled filing-search matches. It is not a real-time EDGAR
feed, so retain a reconciliation path through match and delivery history.

## 1. Configure delivery in the dashboard

An organization administrator uses the signed-in
[Delivery dashboard](https://secapi.ai/app/webhooks) to create a public-HTTPS
endpoint, subscribe it to `monitor.match`, and create a narrowly scoped monitor
with a query, form filter, and ticker list. These are human-authenticated
control-plane actions; an API key cannot create or change them.

Save the dashboard-issued endpoint ID, monitor ID, and signing secret in the
receiver's secret store.

## 2. Preview the saved search

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/monitors/MONITOR_ID/matches?limit=10"
```

The preview shows current saved-search matches; it does not prove immediate
future delivery.

## 3. Verify and recover

Verify signatures against the raw body before parsing JSON, reject stale
timestamps, and make writes idempotent by event ID or filing accession number.
Use the dashboard to send a test event. It emits `webhook.test`, not a
production `monitor.match` event.

- Inspect `GET /v1/webhook_endpoints/{webhook_id}/deliveries`.
- Replay only when duplicates are safe.
- Reconcile `GET /v1/monitors/{monitor_id}/matches` after delivery failures.

A match identifies a disclosure for review; it does not determine materiality.
