---
title: "Analyze Insider Transactions"
description: "Retrieve normalized Forms 3, 4, and 5 transaction records, distinguish supported filters from investment interpretation, and retain filing evidence."
---

Insider records are disclosures, not trading signals. Use them to build a review queue: retain the Form 4 transaction code, security title, owner relationship, accession number, and filing URL before drawing a conclusion.

## Prerequisites

- An API key exported as `SECAPI_API_KEY`.
- `curl` and `jq`, or Node.js 18+.
- A ticker or CIK. This example uses `AAPL`.

```bash
export SECAPI_API_KEY="your_api_key"
```

## 1. List recent Form 4 transactions

`GET /v1/insiders` returns normalized records from Forms 3, 4, and 5. Use `forms=4` to limit this request to Form 4 filings.

```bash
curl --fail-with-body \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/insiders?ticker=AAPL&forms=4&limit=25" \
  | jq '{data, hasMore, nextCursor, requestId, traceparent}'
```

**Result:** a paginated `data` array. Preserve each transaction's owner, filing date, transaction code, shares, price/value when present, and the accession/filing URL evidence returned by the record or its provenance.

## 2. Filter a review queue by transaction code

The supported filter is `transaction_code`, not a generic purchase/sale flag. For example, Form 4 code `P` denotes an open-market or private purchase; use `S` to review sales. Keep the raw code because the SEC code, not an inferred label, is the source record.

```bash
curl --fail-with-body \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/insiders?ticker=JPM&forms=4&transaction_code=P&date_from=2025-01-01&limit=50" \
  | jq '{data, hasMore, nextCursor, requestId}'
```

Other supported filters include `owner_name`, `owner_cik`, `security_title`, `date_from`, `date_to`, and `cursor`. `limit` is 1-50; advance with the returned `nextCursor` while `hasMore` is true.

## Node.js implementation

```js
const url = new URL("https://api.secapi.ai/v1/insiders");
url.search = new URLSearchParams({
  ticker: "JPM",
  forms: "4",
  transaction_code: "P",
  date_from: "2025-01-01",
  limit: "50",
}).toString();

const response = await fetch(url, {
  headers: { "x-api-key": process.env.SECAPI_API_KEY ?? "" },
});
if (!response.ok) throw new Error(`SEC API ${response.status}: ${await response.text()}`);

const payload = await response.json();
console.log(JSON.stringify({
  count: payload.data?.length ?? 0,
  hasMore: payload.hasMore,
  nextCursor: payload.nextCursor,
  requestId: payload.requestId,
  trades: (payload.data ?? []).map((trade) => ({
    owner: trade.ownerName ?? trade.insiderName ?? null,
    transactionCode: trade.transactionCode ?? null,
    transactionDate: trade.transactionDate ?? null,
    shares: trade.transactionShares ?? null,
    value: trade.transactionValue ?? null,
    accessionNumber: trade.accessionNumber ?? trade.provenance?.accessionNumber ?? null,
    filingUrl: trade.filingUrl ?? trade.provenance?.filingUrl ?? null,
  })),
}, null, 2));
```

## Common failure modes

- `401` or `403`: the key is missing, invalid, or not entitled for the route.
- No records: check the ticker, date range, and `forms=4`; no result is not evidence that no insider activity occurred outside the requested window.
- Duplicate or partial analysis: paginate with `nextCursor`, then deduplicate by accession number and transaction identity in your own store.
- Misleading signal: option exercises, gifts, tax withholding, and planned sales can appear alongside discretionary open-market activity. Review the filing, transaction code, ownership fields, and security title.

## Production notes

Persist the raw record plus `requestId`, `traceparent`, accession number, and filing URL. Use an idempotent ingestion key and a scheduled backfill window, because filings and corrections can arrive after a first poll. Keep alerts descriptive and evidence-linked; do not automatically convert transactions into buy/sell recommendations.
