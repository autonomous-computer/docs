---
title: "Build an SEC Filing Monitor"
description: "Poll supported filing search endpoints, send idempotent alerts, and retain the filing provenance an analyst needs to investigate each event."
---

This monitor polls for new 8-K and 8-K/A filings for a watchlist. It uses the public HTTP API and Node.js 18+ only, so it can run in cron, a container scheduler, or a serverless job without a product-specific CLI or agent runtime.

## Prerequisites

- Node.js 18+ and an API key exposed to the job as `SECAPI_API_KEY`.
- A durable store for seen accession numbers. The local JSON file below is a development example, not a multi-worker database.
- An optional Slack incoming-webhook URL in `SLACK_WEBHOOK_URL`.

```bash
export SECAPI_API_KEY="your_api_key"
export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/your/webhook/url"
mkdir sec-filing-monitor && cd sec-filing-monitor
printf '{"seen":[]}' > state.json
```

## 1. Use the supported filings query

`GET /v1/filings` supports `ticker`, `form` or comma-separated `forms`, `date_from`, `date_to`, `limit`, `cursor`, and `sort=filing_date_desc|filing_date_asc`. This request is a useful manual smoke test.

```bash
curl --get --fail-with-body \
  -H "x-api-key: $SECAPI_API_KEY" \
  --data-urlencode "ticker=AAPL" \
  --data-urlencode "forms=8-K,8-K/A" \
  --data-urlencode "limit=10" \
  --data-urlencode "sort=filing_date_desc" \
  "https://api.secapi.ai/v1/filings" \
  | jq '{data, hasMore, nextCursor, requestId, traceparent}'
```

## 2. Run an idempotent monitor

Create `monitor.mjs`. The script records accession numbers only after it has successfully handled an item; replace the JSON state with a transactional store before running more than one worker.

```js
import { readFile, writeFile } from "node:fs/promises";

const tickers = ["AAPL", "MSFT", "NVDA"];
const forms = "8-K,8-K/A";
const statePath = new URL("./state.json", import.meta.url);
const apiKey = process.env.SECAPI_API_KEY;
if (!apiKey) throw new Error("SECAPI_API_KEY is required");

const state = JSON.parse(await readFile(statePath, "utf8"));
const seen = new Set(state.seen ?? []);
const alerts = [];

for (const ticker of tickers) {
  const query = new URLSearchParams({ ticker, forms, limit: "50", sort: "filing_date_desc" });
  const response = await fetch(`https://api.secapi.ai/v1/filings?${query}`, {
    headers: { "x-api-key": apiKey },
  });
  if (!response.ok) throw new Error(`${ticker}: ${response.status} ${await response.text()}`);

  const payload = await response.json();
  for (const filing of payload.data ?? []) {
    const accessionNumber = filing.accessionNumber ?? filing.accession_number;
    if (!accessionNumber || seen.has(accessionNumber)) continue;
    alerts.push({
      ticker,
      form: filing.form,
      filingDate: filing.filingDate ?? filing.filedAt ?? null,
      accessionNumber,
      filingUrl: filing.filingUrl ?? filing.provenance?.filingUrl ?? null,
      requestId: payload.requestId,
    });
    seen.add(accessionNumber);
  }
}

if (process.env.SLACK_WEBHOOK_URL && alerts.length) {
  const text = alerts.map((item) =>
    `${item.ticker} ${item.form} ${item.filingDate ?? ""} ${item.accessionNumber} ${item.filingUrl ?? ""}`.trim(),
  ).join("\n");
  const slack = await fetch(process.env.SLACK_WEBHOOK_URL, {
    method: "POST", headers: { "content-type": "application/json" }, body: JSON.stringify({ text }),
  });
  if (!slack.ok) throw new Error(`Slack ${slack.status}: ${await slack.text()}`);
}

await writeFile(statePath, JSON.stringify({ seen: [...seen].slice(-5_000) }, null, 2));
console.log(JSON.stringify({ checked: tickers.length, newFilings: alerts.length, alerts }, null, 2));
```

Run it once with `node monitor.mjs`. A successful result reports the number of issuers checked and new filing records with their accession numbers and source URLs. The first run may alert on existing filings because the state starts empty; seed or backfill the state intentionally.

## Common failure modes

- `401` or `403`: the scheduler did not receive a valid API key or route entitlement.
- Invalid results: use `forms`, not `form_type`, and `filing_date_desc`, not an unsupported sort token.
- Duplicate alerts: use the accession number as the durable idempotency key; store alert delivery separately if sending can fail after state changes.
- Missed events: paginate with `nextCursor` when `hasMore` is true, overlap polling windows, and reconcile against a recent date range.
- Time-zone mistakes: schedule by a named time zone where your scheduler supports it; do not assume a fixed UTC offset for US market time.

## Production notes

Keep secrets in the scheduler's secret store, retry transient HTTP failures with bounded exponential backoff, and emit metrics for polls, pages, new filings, and delivery failures. Retain `requestId`, `traceparent`, accession number, filing URL, query filters, and alert status. Alerting identifies new disclosures; it does not classify their materiality or replace analyst review.
