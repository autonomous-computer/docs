---
title: "Monitor 13F Holdings Changes"
description: "Retrieve a manager's latest 13F report, compare its two latest reports, and preserve the SEC filing evidence needed for an ownership workflow."
---

Form 13F reports are delayed, manager-reported snapshots. They can help an investor research disclosed positioning, but they do not show real-time holdings, every asset class, or intent. Keep each source report's date, accession number, and filing URL with the analysis that uses it.

## Prerequisites

- An API key exported as `SECAPI_API_KEY`.
- `curl` and `jq`, or Node.js 18+.
- The reporting manager's CIK. This example uses `0001067983`.

```bash
export SECAPI_API_KEY="your_api_key"
```

## 1. Retrieve the latest report

`GET /v1/owners/13f` requires `cik`. It also accepts `limit` (1-50), plus optional `reportDate` and `filingDate` selectors.

```bash
curl --get --fail-with-body \
  -H "x-api-key: $SECAPI_API_KEY" \
  --data-urlencode "cik=0001067983" \
  --data-urlencode "limit=20" \
  "https://api.secapi.ai/v1/owners/13f" \
  | jq '{managerName, reportDate, filingDate, accessionNumber, filingUrl, holdings, requestId, traceparent, provenance}'
```

**Result:** the most recent normalized report selected for that manager, including a holdings collection. A holding's `valueUsdThousands` is in thousands of US dollars; multiply it by 1,000 only for display or a deliberately chosen storage unit.

## 2. Compare the two latest reports

`POST /v1/owners/13f/compare` takes a JSON body with `cik` and optional `limit` (1-50). It compares the latest two reports available to the route; it is not a request for arbitrary historical quarters.

```bash
curl --fail-with-body \
  -X POST \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "content-type: application/json" \
  --data '{"cik":"0001067983","limit":50}' \
  "https://api.secapi.ai/v1/owners/13f/compare" \
  | jq '{currentFilingDate, previousFilingDate, rows, requestId, traceparent, provenance}'
```

The comparison response identifies changes between the reports identified by `currentFilingDate` and `previousFilingDate`. Preserve those dates and the reported share/value fields; position changes can reflect corporate actions, fund structure, reporting conventions, or a genuine investment decision.

## Node.js implementation

```js
const response = await fetch("https://api.secapi.ai/v1/owners/13f/compare", {
  method: "POST",
  headers: {
    "x-api-key": process.env.SECAPI_API_KEY ?? "",
    "content-type": "application/json",
  },
  body: JSON.stringify({ cik: "0001067983", limit: 50 }),
});
if (!response.ok) throw new Error(`SEC API ${response.status}: ${await response.text()}`);

const payload = await response.json();
console.log(JSON.stringify({
  requestId: payload.requestId,
  traceparent: payload.traceparent,
  provenance: payload.provenance ?? null,
  currentFilingDate: payload.currentFilingDate ?? null,
  previousFilingDate: payload.previousFilingDate ?? null,
  rows: (payload.rows ?? []).map((row) => ({
    issuer: row.issuer ?? null,
    cusip: row.cusip ?? null,
    previousShares: row.previousShares ?? null,
    currentShares: row.currentShares ?? null,
    deltaShares: row.deltaShares ?? null,
    status: row.status ?? null,
  })),
}, null, 2));
```

## 3. Resolve individual filing identities when needed

The comparison response does not return both report accessions. List the manager's recent 13F filings when you need accession-level identity, then join the list to the comparison using the filing dates. `since` is an inclusive filing-acceptance timestamp for incremental polling, and `limit` returns up to 50 records from the route's recent filing scan. The route does not support cursor pagination, so use it for recent or incremental retrieval rather than an exhaustive manager-history backfill.

```bash
curl --get --fail-with-body \
  -H "x-api-key: $SECAPI_API_KEY" \
  --data-urlencode "cik=0001067983" \
  --data-urlencode "since=2025-01-01T00:00:00Z" \
  --data-urlencode "limit=50" \
  "https://api.secapi.ai/v1/owners/13f/filings" \
  | jq '{data, requestId, traceparent}'
```

The same request shape is available at `GET /v1/owners/13f/filings?cik=0001067983&since=2025-01-01T00:00:00Z&limit=50`.

## Common failure modes

- `400 missing_cik`: supply the manager CIK, not a portfolio company ticker.
- `400` for `limit`: it must be a positive integer no greater than 50.
- No comparison: the route needs two available reports. Use `GET /v1/owners/13f/filings?cik=...&since=...&limit=50` to inspect recent or incremental filing availability and resolve accession identities; it is not a cursor-paginated full-history export.
- Incorrect timestamps: distinguish the report's quarter-end date from the SEC filing date. A 13F is reported after the period it describes.
- False precision: do not infer intraperiod trades, total fund exposure, or conviction from one reported line item.

## Production notes

Run after expected 13F filings arrive, retain raw reports and comparisons with request metadata, and deduplicate raw reports by filing accession. Use a database rather than a local file for multi-worker jobs. Show analysts both source reports before escalating a change, and label the data as disclosed, delayed institutional holdings rather than current ownership.
