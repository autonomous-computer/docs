Set `SECAPI_API_KEY` before running an example. Each quickstart answers one
question and preserves the source trail needed to review the result.

## Compare a manager's latest 13F reports

```bash
curl --fail-with-body -X POST "https://api.secapi.ai/v1/owners/13f/compare" \
  -H "x-api-key: $SECAPI_API_KEY" -H "content-type: application/json" \
  -d '{"cik":"0001067983","limit":25}'
```

The response compares the two reports selected by the service. Retain report
dates, position identifiers, and `requestId`. A 13F is historical disclosure,
not a complete or current portfolio.

## Retrieve a filing section

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K"
```

The result includes extracted text and filing provenance. An unavailable or
empty section is a review state, not evidence that the issuer has no risks.

## Find filing passages by topic

```bash
curl --get --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  --data-urlencode "q=supply chain disruption" \
  --data-urlencode "ticker=AAPL" --data-urlencode "form=10-K" \
  "https://api.secapi.ai/v1/search/semantic"
```

Search rows are candidates with citations. Ranking is not a materiality finding;
inspect the cited filing before using a passage in a decision.

## Create a monitor delivery path

In the signed-in [Delivery dashboard](https://secapi.ai/app/webhooks), create a
public-HTTPS webhook endpoint, subscribe it to `monitor.match`, and create a
narrow saved monitor for the required query, forms, and tickers. These are
human-authenticated organization control-plane actions; do not use an API key
or an autonomous agent to create or change them.

Store the dashboard-issued signing secret securely, verify raw-body signatures,
and make the consumer idempotent. Monitor delivery is scheduled; reconcile
matches and delivery history with machine-read API requests.
