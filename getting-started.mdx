---
title: "Make your first SEC API request"
description: "Resolve an issuer and retrieve a source-backed filing with one API key"
---

Make one request that you can use in a real application: resolve an issuer, then retrieve its latest annual filing. Keep the returned accession number, filing URL, and request ID with any result you publish or pass to another system.

## Set your API key

Create an account at [secapi.ai/signup](https://secapi.ai/signup), then create an API key from the signed-in dashboard before exporting it in the shell that will call the API:

```bash
export SECAPI_API_KEY="secapi_..."
```

SEC API machine credentials use `x-api-key`. Do not send an API key as `Authorization: Bearer`.

## Resolve an issuer

Resolve Apple before you request a filing. The response gives you the CIK and canonical identity details that work across SEC API endpoints.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"
```

## Expected response

The exact `id` and `requestId` vary. Confirm that the response identifies Apple and includes CIK `0000320193`.

```json
{
  "object": "entity",
  "ticker": "AAPL",
  "cik": "0000320193",
  "name": "Apple Inc.",
  "requestId": "req_..."
}
```

`view=agent` is a compact projection for code or agent workflows. Use the endpoint's default response when you need the full metadata.

## 2. Retrieve a filing you can cite

Request Apple's latest 10-K using the same key:

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

The filing changes when Apple submits a new annual report. Store the returned `accessionNumber`, `filingDate`, `form`, `filingUrl`, and `requestId` with your downstream result. They identify the specific SEC filing behind a claim.

## When a request fails

For `401 missing_api_key`, make sure the key is set in the current shell and sent through `x-api-key`. For `401 authentication_failed`, create a replacement key in the dashboard and update the calling environment. For `402` or `429`, inspect the response before retrying; a bounded retry is appropriate for transient failures, not for a quota or billing stop.

Read [Troubleshooting](/troubleshooting) for error-specific next steps and [API conventions](/api-conventions) for versioning, pagination, and response metadata.
