---
title: "Make your first SEC API request"
description: "Authenticate a server-side request, retrieve a filing, and keep a citation-ready source record"
---

Your first integration should produce a filing artifact you can inspect and cite. Start with an API key in a server, worker, notebook, or terminal; do not put it in browser code.

## Create an API key

Create an account at [secapi.ai/signup](https://secapi.ai/signup), create an API key in the signed-in [dashboard](https://secapi.ai/app), and expose it only to the calling process:

```bash
export SECAPI_API_KEY="secapi_..."
```

<Warning>
Machine requests authenticate with `x-api-key`. Do not send an API key as `Authorization: Bearer`, put it in a URL, or commit it to a repository.
</Warning>

## Retrieve a filing artifact

Ask for one known issuer and form. This request returns the current latest matching filing, so the accession number can change after a new filing.

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

Before summarizing or storing the result, retain these fields when the response provides them:

- `accessionNumber`
- `form`
- `filingDate`
- `filingUrl`
- `requestId`

Those fields are the minimum filing citation artifact: identify the source as `Apple Inc. 10-K, filed <filingDate>, accession <accessionNumber>, <filingUrl>`. Keep any returned `provenance`, `freshness`, `materialization`, warning, and trace fields alongside it; availability varies by route and response view.

## Resolve before broader work

For a workflow that starts from a name, ticker, CIK, or identifier, resolve the entity first and carry the returned identity forward:

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"
```

Then choose an interface based on where the work lives:

| Interface | Use it when | Next step |
| --- | --- | --- |
| REST | You own HTTP, serialization, retries, and deployment. | [First request flows](/first-request-flows) |
| SDK | Your application is JavaScript, Python, Go, or Rust. | [Libraries and SDKs](/libraries-and-sdks) |
| CLI | You need a terminal check or shell automation. | [CLI](/cli) |
| MCP | An MCP client should discover and call SEC API tools. | [Install MCP](/mcp-install) |

## When a request fails

`401 missing_api_key` means the process did not send an API key; `401 authentication_failed` means the key is not accepted. For `402`, stop and resolve the billing or entitlement condition in the response. For `429`, honor `Retry-After` when it is supplied and reduce concurrent work. Preserve the response's request identifier for support or audit work.

See [API conventions](/api-conventions) for request behavior and [Troubleshooting](/troubleshooting) for error-specific guidance.
