---
title: "Make your first API request"
description: "Authenticate with an SEC API key and resolve Apple to its SEC identity in one request."
---

Use one authenticated request to confirm that your API key, headers, and client can reach SEC API. This takes about three minutes.

## Before you start

You need:

1. An [SEC API account](https://secapi.ai/signup)
2. An API key from the dashboard
3. A shell with `curl`

Export the key in the shell that will make the request:

```bash
export SECAPI_API_KEY="secapi_..."
```

<Warning>
Send machine API keys in the `x-api-key` header. Do not use `Authorization: Bearer`; bearer tokens are for WorkOS browser sessions, not SEC API keys.
</Warning>

## Make the request

Resolve Apple to the identifiers SEC API uses across filings, facts, ownership records, and other datasets:

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"
```

`secapi-version` pins your client to a dated API contract. `view=agent` returns the compact response shape.

## Expected response

The entity ID and request ID will differ, but a successful response has this shape:

```json
{
  "object": "entity",
  "id": "cent_f3913349312cbf5bfd60ecdb",
  "ticker": "AAPL",
  "cik": "0000320193",
  "name": "Apple Inc.",
  "primaryIdentifiers": [
    {
      "type": "ticker",
      "value": "AAPL"
    },
    {
      "type": "cik",
      "value": "0000320193"
    }
  ],
  "matchConfidence": 1,
  "matchBasis": "ticker",
  "requestId": "req_2ZK8Q1W9F4M6P7R3"
}
```

You are ready to continue when the response identifies Apple, returns CIK `0000320193`, and includes a `requestId`.

## What to keep

- **`id`, `ticker`, and `cik`** identify the issuer in later requests.
- **`matchConfidence` and `matchBasis`** explain why this record matched your input.
- **`requestId`** identifies the API call for logs, support, and debugging.

The compact `view=agent` response intentionally omits larger metadata envelopes. Endpoints that return filing-derived data can also include source, freshness, materialization, or trace fields. Keep those fields when they are present; do not infer them from this entity-resolution response.

## If the request fails

### `401 missing_api_key`

Confirm that the environment variable is set in the same shell:

```bash
test -n "$SECAPI_API_KEY" && echo "SECAPI_API_KEY is set"
```

Then confirm the request uses `x-api-key`.

### `401 authentication_failed`

Remove any `Authorization: Bearer` header and send the SEC API key through `x-api-key` instead.

### `429` or a budget error

Read the response code and limit details before retrying. An agent should stop or back off rather than repeat the same call without a bound.

### The service appears unavailable

Check [service health](https://api.secapi.ai/healthz), [readiness](https://api.secapi.ai/readyz), and [SEC API status](https://status.secapi.ai). These are troubleshooting tools, not prerequisites for a normal first request.

For response envelopes, versioning, errors, and request metadata, read [API conventions](/api-conventions). For a broader failure checklist, use [Troubleshooting](/troubleshooting).
