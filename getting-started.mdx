---
title: "Make your first SEC API request"
description: "Use one API key to resolve an issuer and retrieve a source-backed 10-K in minutes."
---

Your first successful request should answer a real research question. In this guide, you will resolve Apple to its SEC identity, then retrieve its latest 10-K with the filing record you need to cite it later.

## Before you start

You need:

1. An [SEC API account](https://secapi.ai/signup)
2. An API key from the dashboard
3. A shell with `curl`

Export the key in the shell that will make the request:

```bash
export SECAPI_API_KEY="secapi_..."
```

<Warning>
Send machine API keys in the `x-api-key` header. Do not use `Authorization: Bearer`; bearer tokens are for WorkOS browser sessions, not SEC API keys.
</Warning>

## 1. Resolve the issuer

Resolve Apple to the identifiers SEC API uses across filings, facts, ownership records, and other datasets:

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"
```

`view=agent` returns the compact response shape. Use a dated `secapi-version` header in production once you have chosen a contract version to pin.

## Expected response

The entity ID and request ID will differ, but a successful response has this shape:

```json
{
  "object": "entity",
  "id": "cent_f3913349312cbf5bfd60ecdb",
  "ticker": "AAPL",
  "cik": "0000320193",
  "name": "Apple Inc.",
  "primaryIdentifiers": [
    {
      "type": "ticker",
      "value": "AAPL"
    },
    {
      "type": "cik",
      "value": "0000320193"
    }
  ],
  "matchConfidence": 1,
  "matchBasis": "ticker",
  "requestId": "req_2ZK8Q1W9F4M6P7R3"
}
```

You are ready to continue when the response identifies Apple, returns CIK `0000320193`, and includes a `requestId`.

## What to keep

- **`id`, `ticker`, and `cik`** identify the issuer in later requests.
- **`matchConfidence` and `matchBasis`** explain why this record matched your input.
- **`requestId`** identifies the API call for logs, support, and debugging.

The compact entity response is deliberately small. The next request is where you begin to receive filing-specific source context.

## 2. Retrieve a filing you can cite

Ask for Apple's latest annual filing in the same compact shape:

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

The exact accession and dates will change as Apple files. Before you use the result in a memo, model, or agent answer, retain the accession number, filing date, form, and source URL that the response supplies. Those fields identify the filing you actually read; a ticker alone does not.

## 3. Pick the next useful action

- Extract a section such as [Risk Factors or MD&A](/api-reference/filings/get-v1-filings-latest-sections-section-key) when you need the disclosure text behind a conclusion.
- Pull [structured statements](/api-reference/statements) when you need filed numbers rather than narrative.
- Use [filing search](/api-reference/filings) when you need a historical set rather than the latest filing.
- Connect [MCP](/mcp-workflows) only after this REST request works; it uses the same machine credential.

## If the request fails

### `401 missing_api_key`

Confirm that the environment variable is set in the same shell:

```bash
test -n "$SECAPI_API_KEY" && echo "SECAPI_API_KEY is set"
```

Then confirm the request uses `x-api-key`.

### `401 authentication_failed`

Remove any `Authorization: Bearer` header and send the SEC API key through `x-api-key` instead.

### `429` or a budget error

Read the response code and limit details before retrying. An agent should stop or back off rather than repeat the same call without a bound.

### The service appears unavailable

Check [SEC API status](https://status.secapi.ai). Use [service health](https://api.secapi.ai/healthz) and [readiness](https://api.secapi.ai/readyz) when you are debugging an integration, not as part of every normal request.

For response envelopes, versioning, errors, and request metadata, read [API conventions](/api-conventions). For a broader failure checklist, use [Troubleshooting](/troubleshooting).
