---
title: "Billing-aware agent runs"
description: "Check the account that will pay for a run, quote repeated work, and stop at a real billing or quota boundary"
---

An agent that can make one request is not necessarily safe to fan out across a watchlist. Before a repeated or expensive job, inspect the organization attached to its API key and record the result alongside the job.

## Check the run's account

```bash
curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  https://api.secapi.ai/v1/billing

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  https://api.secapi.ai/v1/limits
```

These calls describe the authenticated organization, not a general price list. `GET /v1/billing/rates` is useful for the public catalog, but it cannot tell an agent whether this organization can perform the next request.

## Quote the repeated operation

Quote the route or meter class before starting a loop. The response reflects the current plan and budget gate; it is not a reservation and does not guarantee a later request will have the same result.

```bash
curl --fail-with-body -sS -X POST \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "content-type: application/json" \
  -d '{"path":"/v1/filings/latest","method":"GET","units":1}' \
  https://api.secapi.ai/v1/billing/quote
```

Save `requestId`, `planKey`, `meterClass`, `units`, `budget`, and `budgetGate` with the run record. Quote one representative unit, then make the fan-out size an explicit operator decision.

## Use a stop rule

Do not retry a refusal until something has changed. Stop, retain the response context, and hand the case back for authorization, billing, budget-gate, hard-cap, or quota errors. For `429`, honor `Retry-After` when it is present and reduce concurrency rather than starting parallel retries.

API keys can read billing state, limits, and quotes. Budget changes use a signed-in organization session. An unattended agent should not receive a human bearer token to raise its own cap.

## Bootstrap a machine without sharing a long-lived key

A human organization member can issue a short-lived, single-use sponsor token through [agent bootstrap tokens](/api-reference/agent/post-v1-agent-bootstrap-tokens). The machine exchanges it at [agent bootstrap](/api-reference/agent/post-v1-agent-bootstrap), which returns an org-scoped API key together with billing, limits, and MCP install metadata.

The sponsor-token secret and issued API-key secret are each sensitive and returned once. Do not put either in logs, task output, source control, or a prompt transcript. Expired, used, invalid, scope-limited, and API-key-limit responses are stop conditions, not reasons to mint credentials in a loop.
