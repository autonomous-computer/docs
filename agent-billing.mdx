---
title: "Agent billing"
description: "Check organization billing state, quote repeated work, and stop cleanly at billing or quota boundaries"
---

An agent can make read requests with an API key, but it should inspect the billing state that governs that same key before it begins repeated work.

```bash
curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/billing"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"
```

These are organization-specific responses. The public `GET /v1/billing/rates` catalog is useful for learning available meter classes, but it does not establish whether this organization can make a later request.

## Quote a repeated step

Use [POST /v1/billing/quote](/api-reference/platform/post-v1-billing-quote) before a repeated workflow. A quote evaluates the current plan and budget gate; it does not reserve capacity or guarantee a later billing state.

```bash
curl --fail-with-body -sS -X POST \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "content-type: application/json" \
  -d '{"path":"/v1/filings/latest","method":"GET"}' \
  "https://api.secapi.ai/v1/billing/quote"
```

Store the quote response and request identifier with the agent's decision. Do not construct a quote from a guessed price, fixed rate, or historical response.

## Stop conditions

Do not loop on a billing or quota refusal. Surface the server response to the operator and stop when it reports a billing requirement, a budget approval requirement, a hard-cap condition, or an AI-query quota condition. For `429`, honor `Retry-After` when supplied and reduce fan-out. For every error, retain the request context returned by the API.

## Keep account changes human-authenticated

API keys can read billing state, limits, and quotes. Billing mutations, including a budget change, use browser-session organization authorization. Do not give an unattended agent a human bearer token to change its own budget; send an authorized person to the signed-in dashboard or the applicable browser-authenticated endpoint.

For agent credential bootstrap, a human organization member can create a sponsor token through [agent bootstrap tokens](/api-reference/agent/post-v1-agent-bootstrap-tokens). The agent exchanges it through [agent bootstrap](/api-reference/agent/post-v1-agent-bootstrap). Treat both the sponsor token and returned API key as secrets, and surface expired, used, invalid, or scope-limited token errors without logging the secret.
