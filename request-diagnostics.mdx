---
title: "Request Diagnostics"
description: "Capture the request identifiers, headers, status, and billing context SEC API support needs to debug a failed REST, SDK, CLI, or MCP request."
---

Use this checklist when a request still fails after the first troubleshooting pass. The goal is to preserve enough context for support or an internal operator to reproduce the request without exposing secrets.

## Capture identifiers

- `Request-Id` response header
- `traceparent` response header, when present
- HTTP method and endpoint path
- UTC timestamp and timezone
- SDK, CLI, MCP client, or direct HTTP caller version
- Organization, workspace, or environment name, but never the API key value

## Capture the response

- HTTP status code
- Error `code`
- Error `message`
- `docsUrl`, when present
- `Retry-After`, quota, and rate-limit headers
- Whether the same key succeeds against `GET /v1/limits`

## Capture billing and quota context

Run the two baseline checks with the same API key:

```bash
curl "https://api.secapi.ai/v1/limits" \
  -H "x-api-key: $SECAPI_API_KEY"

curl "https://api.secapi.ai/v1/billing" \
  -H "x-api-key: $SECAPI_API_KEY"
```

For `402` and `429` responses, include quota headers, billing state, budget state, and the approximate retry schedule your application used. Do not keep retrying quota or hard-cap errors in a loop.

## Redact before sharing

- Redact `x-api-key`, bearer tokens, cookies, and organization secrets.
- Redact unreleased filing text, private portfolio holdings, customer names, and proprietary prompts unless support explicitly asks for a narrowed reproduction.
- Keep accession numbers, endpoint paths, request IDs, status codes, and error codes visible.
