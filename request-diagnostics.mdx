---
title: "Request diagnostics"
description: "Capture a minimal, secret-free request record to diagnose SEC API REST, SDK, CLI, or MCP failures"
---

Capture the response before retrying. A status code alone rarely distinguishes an invalid request, an account limit, a transient failure, and a partial data result.

```bash
curl -i -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"
```

The `-i` flag prints response headers. Retain the result privately; never include the API key in a ticket, log excerpt, or screenshot.

## Capture this record

| Field | Why it matters |
| --- | --- |
| HTTP method and endpoint path | Identifies the route and operation |
| UTC timestamp | Places the request in time without timezone ambiguity |
| HTTP status and error `code` | Distinguishes request, authentication, billing, and availability failures |
| Error `message`, `hint`, and `docsUrl` when returned | Preserves route-specific guidance |
| `Request-Id` and `traceparent` when present | Correlates the request with SEC API diagnostics |
| `Retry-After` and any quota or billing headers | Explains when and whether a retry is appropriate |
| SDK, CLI, MCP client, or HTTP library version | Makes client-specific failures reproducible |

For a request body, retain a redacted version plus the names of relevant headers. Include the response view, pagination cursor, and query parameters when they affect the result.

## Check the credential context

Run these requests with the same API key that made the failing request:

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"

curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/billing"
```

For `402` and `429` responses, include the returned billing or quota context and the retry schedule your application used. Do not retry a billing, budget, or hard-cap response in a tight loop. For `429`, honor `Retry-After` when it is returned.

## Correlate requests safely

You can send `x-request-id` to associate an API call with your own job or log record. SEC API returns it as the request correlation ID. It is not a billing idempotency key; do not rely on it to deduplicate work or charges.

## Redact before sharing

- Remove `x-api-key`, bearer tokens, cookies, and other credentials.
- Remove customer data, private holdings, unreleased text, and proprietary prompts unless a narrowed reproduction requires them.
- Keep endpoint paths, status codes, request IDs, error codes, and the smallest useful response excerpt.
