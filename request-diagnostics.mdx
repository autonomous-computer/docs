---
title: "Request diagnostics"
description: "Capture a concise, secret-free record for an SEC API REST, SDK, CLI, or MCP failure"
---

Capture the failing response before retrying. A status code alone cannot reliably distinguish a malformed request, rejected credential, billing condition, rate limit, or transient service failure.

## Make one inspectable request

```bash
curl -i -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"
```

`-i` includes response headers. Store the result privately and never include the API key in a ticket, log excerpt, or screenshot.

## Capture this minimum record

| Capture | Why |
| --- | --- |
| HTTP method and endpoint path | Identifies the operation without exposing secrets |
| UTC timestamp, status, and error `code` | Places the failure and separates major failure classes |
| Returned `message`, `hint`, and `docsUrl` | Preserves route-specific guidance when supplied |
| `Request-Id` and `traceparent` | Correlates the request when present |
| `Retry-After` and returned quota or billing headers | Explains whether and when a retry is appropriate |
| SDK, CLI, MCP client, or HTTP-library version | Makes a client-specific failure reproducible |

For a request body, preserve a redacted copy and the names of relevant headers. Include query parameters, response view, and pagination cursor when they affect the result.

## Classify before retrying

- `400` or another validation response: correct the request from the endpoint or tool schema.
- `401` or `403`: verify the credential context. API data requests use `x-api-key`; a dashboard bearer session is not a replacement.
- `402`: stop and resolve the billing or entitlement condition described by the response.
- `429`: honor `Retry-After` when returned and reduce concurrency before retrying.
- `5xx` or transport failure: use bounded backoff and keep the captured request ID.

Do not turn a billing, hard-cap, or validation failure into a tight retry loop.

## Verify the same credential context

Run these with the same key used by the failing workload:

```bash
curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/billing"
```

For an MCP failure, include the client name and version, server URL, tool or method name, and returned request identifier. For SDK and CLI failures, include their versions and the smallest redacted reproduction.

## Redact before sharing

Remove `x-api-key`, bearer tokens, cookies, and other credentials. Remove customer data, private holdings, unreleased text, and proprietary prompts unless a narrowed reproduction needs them. Keep the endpoint path, status, request ID, error code, and the smallest useful response excerpt.
