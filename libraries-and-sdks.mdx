## Verify an installation with one workflow

After installing an SDK or CLI, run these four operations for a known issuer:

1. Resolve `AAPL`.
2. Retrieve its latest `10-K`.
3. Retrieve Item `item_1a` from that latest filing.
4. Retrieve one annual all-statements bundle.

The JavaScript method names are `resolveEntity`, `latestFiling`, `latestSection`, and `allStatements`. The Python equivalents are `resolve_entity`, `latest_filing`, `latest_section`, and `all_statements`. In the CLI, use `entities resolve`, `filings latest`, `sections get`, and `statements get` with `--statement all`.

This verifies credentials, package installation, egress, and response shape with data you can recognize. The latest filing, extracted section text, and statement values change over time. Assert against response shape and source identifiers, not against one historical accession number.

## Operate the client safely

Store the API key in the runtime's secret store and expose it as `SECAPI_API_KEY`; do not embed it in frontend code or commit it. Let one layer own retries. On `429`, honor `Retry-After` and reduce concurrency rather than adding parallel retry loops.

Keep `requestId` with every result. For filing-derived output, retain the accession number, filing date, and source URL when present. Preserve `provenance`, `freshness`, `materialization`, and trace fields when the endpoint returns them.

SDKs handle documented transient failures for safe reads. Your application should still set its own request deadlines, preserve request IDs, and avoid layering unbounded retries around the client. Only retry a write when your application can safely repeat it.
