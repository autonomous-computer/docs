title: "Webhook and Stream Workflows"
description: "Configure organization delivery in the dashboard, verify signed SEC API events, and inspect delivery history with an API key."
---

Use webhooks or streams when your application should react to SEC API events. The configuration boundary is deliberate: create endpoints and subscriptions in the signed-in [SEC API dashboard](https://secapi.ai/app/webhooks), then use API keys for read-only data, evidence, and delivery inspection.

## Configure in the dashboard

The dashboard is required to create or change webhook endpoints, stream subscriptions, signing secrets, test deliveries, replays, and monitor delivery settings. Those actions require a WorkOS browser session and return `human_auth_required` when called with an API key.

For a filing monitor, register a public HTTPS endpoint, subscribe it to `monitor.match`, save the displayed signing secret, and then create the monitor that should feed the endpoint.

## Verify every event

Verify the exact raw request body and the `x-secapi-signature` header before parsing JSON. Check the event type explicitly, preserve the source URL and request ID, and treat a delivery as a notification to review evidence rather than a final conclusion.

## Inspect delivery data with an API key

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/delivery/events?type=monitor.match&limit=25"

curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/webhook_endpoints/WEBHOOK_ID/deliveries?limit=10"
```

Use `GET /v1/event_types?status=public_emitting` as the source of truth for public event names. Return to the dashboard for a configuration change, secret rotation, replay, or test delivery.
