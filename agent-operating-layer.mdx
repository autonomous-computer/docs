---
title: "Building SEC API agents"
description: "Build a bounded SEC API agent workflow with source evidence, billing checks, and explicit failure handling"
---

Build an SEC API agent as a small, inspectable workflow: resolve the source record, request only the derived surface needed, preserve response evidence, and stop when the API reports an authorization, billing, quota, or capability boundary.

## Start with one bounded request

Give the agent a specific issuer, filing family, date range, or question. Begin with the route that returns the source record, then make a second request only when the source response supports it.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

The `agent` view is route-specific and can omit fields from the default response. Check the endpoint reference before relying on a field, and retain the accession number, filing URL, period, provenance, warnings, and route-specific capability or freshness fields that are returned.

## Give the agent a failure policy

An agent should expose rather than hide these conditions:

| Condition | Required behavior |
| --- | --- |
| `not_disclosed`, `unavailable`, `degraded`, or route-specific missing evidence | Return the condition and usable source evidence; do not invent a substitute. |
| `400` | Correct the endpoint parameters or body from the reference. |
| `401` or `403` | Stop and repair the credential or authorization. |
| `402` | Stop and resolve the stated billing or entitlement condition. |
| `429` | Honor `Retry-After` when present; use bounded backoff without retry fan-out. |
| `5xx` | Retry once with bounded backoff, then retain the request ID. |

Keep the endpoint, parameters, `Request-Id`, `traceparent` when present, retrieval time, and returned source evidence in the agent's output. See [API conventions](/api-conventions) and [Freshness and source evidence](/freshness-and-trust).

## Control repeated work

Before a repeated or billable step, call `GET /v1/billing` and [GET /v1/limits](/api-reference/platform/get-v1-limits) with the agent's API key. Use [POST /v1/billing/quote](/api-reference/platform/post-v1-billing-quote) before repeating a billable workflow. A quote reflects the current plan and budget gate; it does not reserve usage or override later limits.

Stop on `billing_budget_approval_required`, `billing_budget_hard_cap_exceeded`, or `ai_query_quota_exceeded`. Do not ask an agent to bypass organization billing controls. Read [Agent billing](/agent-billing) for the public contract and next actions.

## Bootstrap an agent credential safely

A human organization member can create a short-lived, single-use sponsor token with a bearer-authenticated request to `POST /v1/agent/bootstrap_tokens`. The agent submits the token to `POST /v1/agent/bootstrap` and receives the response contract's API key, organization, billing, limits, and installation metadata.

The self-serve flow is limited to the documented `read:sec` scope. A token can be invalid, expired, already used, scope-invalid, or blocked by the active API-key limit; surface that error to the operator without logging the token or issued key. See [agent bootstrap tokens](/api-reference/agent/post-v1-agent-bootstrap-tokens) and [agent bootstrap](/api-reference/agent/post-v1-agent-bootstrap).
