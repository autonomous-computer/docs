---
title: "Give an Agent Read-Only SEC API Access"
description: "Issue a short-lived sponsor token from a signed-in session, exchange it once for an agent API key, and keep control-plane changes human-owned."
---

Use agent bootstrap when a human operator needs to give an ephemeral agent runtime its own SEC API credential. The credential is limited to the self-serve read scope. It can retrieve data through REST or hosted MCP; it cannot configure billing, monitors, webhook destinations, or stream subscriptions.

## Choose the right credential

Use a normal API key for a service you operate. Use an agent bootstrap token when a signed-in organization member is sponsoring a separate runtime and does not want to place a long-lived key in that runtime.

The sponsor-token request requires a WorkOS-authenticated organization session. The exchange is intentionally one-time; keep both the token and the resulting API key out of prompts, browser code, logs, and source control.

## Sponsor a bounded agent job

From your authenticated server or dashboard-backed session, create a short-lived token for one concrete job. The current contract accepts only the self-serve read scope.

```bash
curl --fail-with-body -X POST "https://api.secapi.ai/v1/agent/bootstrap_tokens" \
  -H "Authorization: Bearer $SECAPI_BEARER_TOKEN" \
  -H "content-type: application/json" \
  -d '{"label":"daily-10-k-research","ttlSeconds":900,"scopes":["read:sec"]}'
```

The response is an `agent_bootstrap_token`. Pass its secret directly to the worker through its secret channel. Do not print the response in a build log.

## Exchange it once in the worker

```bash
curl --fail-with-body -X POST "https://api.secapi.ai/v1/agent/bootstrap" \
  -H "content-type: application/json" \
  -d '{"token":"'$SECAPI_BOOTSTRAP_TOKEN'","label":"daily-10-k-research","scopes":["read:sec"]}' \
  | jq '{object, requestId, organization: .organization.id, apiKey: .apiKey.id, sponsorToken}'
```

Expect `201` and `object: "agent_bootstrap"`. Store the returned API-key secret only in the running worker, then use it as `x-api-key` for REST or the hosted [MCP server](/mcp-install). Retain `requestId`, the token id, and the job label with your run record; never retain either secret.

## Handle failure deliberately

- `404 bootstrap_token_invalid`: the token is malformed or unavailable. Request a new token; do not guess or retry a different scope.
- `409 bootstrap_token_already_used`: another worker consumed it. Stop the duplicate worker and issue a new token only for the intended job.
- `410 bootstrap_token_expired`: sponsor a new short-lived token after checking the queue delay.
- `400 bootstrap_token_scope_invalid`: the requested scope is outside the read-only contract.
- `402 api_key_limit_reached`: revoke or rotate an unused key from the signed-in dashboard before retrying the exchange.

## Production boundary

Treat the bootstrap exchange as credential provisioning, not as agent authorization to change your organization. A person must configure [monitors](/monitors), [webhooks](/webhook-stream-workflows), streams, billing, and key lifecycle in the signed-in dashboard. For each agent run, log the job input, route/tool names, `requestId`, `traceparent` when returned, and filing identifiers used in its answer.
