---
title: "Building SEC API agents"
description: "Design a bounded SEC API agent that retains source evidence and stops at real authorization, billing, and capability boundaries"
---

An SEC API agent should produce a reviewable trail, not just an answer. Give it a bounded issuer and question, retrieve the source record, preserve the evidence, and make a second call only when the first supports it.

## Start with a filing, not a conclusion

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

Keep the returned `accessionNumber`, `form`, `filingDate`, `filingUrl`, and `requestId`. Retain route-provided provenance, freshness, materialization, warnings, and trace data when available. Response views are route-specific, so use the endpoint reference before relying on a field for caching or audit policy.

## Give the agent an explicit policy

| Condition | Agent behavior |
| --- | --- |
| Missing, unavailable, degraded, or route-specific evidence state | Return that state and the usable source record. Do not fabricate a substitute. |
| `400` | Correct arguments from the documented endpoint or discovered MCP schema. |
| `401` or `403` | Stop and repair the credential or authorization. |
| `402` | Stop and surface the billing or entitlement condition. |
| `429` | Honor `Retry-After` when present and use bounded backoff without retry fan-out. |
| `5xx` | Apply the calling surface's bounded transient-failure policy and keep the request ID. |

Record the endpoint or tool name, arguments, retrieval time, request identifier, and source artifact with the final answer. See [API conventions](/api-conventions) and [Troubleshooting](/troubleshooting).

## Choose the right execution surface

- REST is the direct HTTP contract when your service owns timeouts and retries.
- SDKs provide language-native clients over that contract.
- The CLI is for terminal inspection and scripts.
- MCP lets an MCP-aware client discover tools and schemas at runtime.

Do not make one surface impersonate another. An MCP client should use `tools.search` and `tools.describe`; an application service should call REST or an SDK; a shell workflow should use CLI commands documented by `secapi <command> --help`.

## Bound repeated work and credential issuance

Read `GET /v1/billing` and `GET /v1/limits` with the agent's API key before repeated work, then use `POST /v1/billing/quote` when a workflow needs a current estimate. Quotes are not reservations. Stop on billing and quota conditions rather than attempting to bypass them; [Agent billing](/agent-billing) describes the boundary.

A human organization member can create a short-lived sponsor token through [agent bootstrap tokens](/api-reference/agent/post-v1-agent-bootstrap-tokens). An agent exchanges that token through [agent bootstrap](/api-reference/agent/post-v1-agent-bootstrap). Do not log the sponsor token or issued API key, and surface token expiry, use, scope, or key-limit failures to the operator.
