---
title: "Migrate from sec-api.io to SEC API"
description: "Dual-run one sec-api.io workflow against SEC API and cut over only when filing identity, response behavior, and evidence pass your acceptance test."
---

Use this guide to replace one sec-api.io production read without treating similarly named endpoints as equivalent contracts. Start with a workflow that has a clear pass/fail result: issuer resolution, filing search, a latest filing, or one section.

## Define the first acceptance test

| Existing need | SEC API route | Acceptance record |
| --- | --- | --- |
| Resolve ticker or CIK | `GET /v1/entities/resolve` | Company name, ticker, CIK, and the persisted identifier. |
| Search filings | `GET /v1/filings` | Form, filing date, accession number, filing URL, and pagination behavior. |
| Get a latest filing | `GET /v1/filings/latest` | Returned filing identity and retrieval time. |
| Read a latest section | `GET /v1/filings/latest/sections/{section_key}` | Filing identity, section key, text availability, and warnings. |
| Read a reported fact | `GET /v1/facts` | Tag, unit, fiscal period, and source filing evidence. |

The API reference, not a successful sample response, defines route parameters and fields. Do not infer broad historical or exhibit availability from one issuer or filing family.

## Verify identity before extraction

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL"

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AAPL&form=10-K&limit=10"
```

For each matched filing, compare ticker, CIK, form, filing date, accession number, and source URL. Retain `requestId` and returned provenance, freshness, materialization, or warning fields when the route provides them.

## Test a bounded extraction

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K"
```

The section is evidence for a specific question, not a substitute for a complete filing. If the current integration uses an arbitrary filing URL, an exhibit, or a specific rendered document, test that exact requirement and keep the old path until it passes.

## Release one reversible change

- Pin `secapi-version` when your client requires a dated version.
- Use `x-api-key` for machine credentials; never put it in a URL or browser bundle.
- Compare nullability, pagination, and source identifiers as well as business fields.
- Honor `Retry-After` for `429` and avoid retrying failures whose condition has not changed.
- Keep a fallback for every unqualified route or filing family.

## Next action

Run the selected read against both services for a defined issuer and form set, then promote only that path when the recorded acceptance test passes.
