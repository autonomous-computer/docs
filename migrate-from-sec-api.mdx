---
title: "Migrate from sec-api.io to SEC API"
description: "Dual-run a sec-api.io filing workflow against SEC API, compare source evidence, and cut over only after the route contract passes your test."
---

Start with the smallest production workflow that has a clear acceptance test: issuer resolution, filing search, a latest filing, or one section. A global URL replacement hides the differences that matter most: identity, pagination, source records, and unavailable states.

## Choose a first workflow

| Existing need | SEC API route | Acceptance record |
| --- | --- | --- |
| Resolve ticker or CIK | `GET /v1/entities/resolve` | Company name, ticker, CIK, and the identifier your app persists. |
| Search filings | `GET /v1/filings` | Form, filing date, accession number, filing URL, and pagination behavior. |
| Get the current filing | `GET /v1/filings/latest` | Returned source identity and retrieval time. |
| Read a latest filing section | `GET /v1/filings/latest/sections/{section_key}` | Filing identity, section key, text availability, and any route warnings. |
| Read a reported financial concept | `GET /v1/facts` | Tag, units, fiscal period, and source filing evidence. |

The API reference is authoritative for route parameters and response fields. Do not infer broad historical availability from a successful test on one issuer or filing family.

## Run an identity-first proof

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AAPL&form=10-K&limit=10&view=agent"
```

For every matched filing, compare ticker, CIK, form, filing date, accession number, and source URL. Save `requestId` and returned provenance, freshness, materialization, or warning fields when present.

## Test a bounded extraction

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&view=agent"
```

Treat the returned section as evidence for a bounded question, not as a substitute for the complete filing. If your current workflow depends on an arbitrary filing URL, an exhibit, or a specific rendered representation, test that exact requirement. Keep the existing path as a temporary fallback until the SEC API route returns the representation and source evidence your application needs.

## Cutover checklist

- Pin `secapi-version` if your production client requires a dated version.
- Send machine credentials through `x-api-key`, never a query parameter or browser bundle.
- Compare nullability, pagination, and source identifiers as well as business fields.
- Honor `Retry-After` for `429`; do not retry authentication, authorization, billing, or hard-cap stops until their condition changes.
- Keep a reversible fallback for any unqualified route or filing family.
- Preserve `Request-Id` and `traceparent` when available for both successful and failed dual-run calls.
