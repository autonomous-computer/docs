## Migration strategy

Do not migrate by replacing every URL in one pass. Pick one production workflow, run it side by side, and compare the evidence your application actually consumes.

<Steps>
  <Step title="Start with issuer and filing identity">
    Map ticker and CIK resolution first. Confirm company identity, form type, accession number, filing date, source URL, and any internal identifier you join on downstream.
  </Step>
  <Step title="Replace the narrowest workflow">
    Move filing search, latest filing retrieval, or one section-extraction path before broader XBRL, ownership, or intelligence workflows.
  </Step>
  <Step title="Keep a fallback for unsupported parity">
    If your current integration depends on arbitrary historical filing URLs or rendered HTML extractor output, keep that specific path on the old provider until the SEC API replacement passes your acceptance test.
  </Step>
</Steps>

## Route mapping

<Warning>
If you depend on arbitrary historical filing URLs or HTML extractor output, keep a temporary fallback path while you finish historical filing-target support. SEC API is stronger on normalized workflows than on raw URL-style extractor parity.
</Warning>

| Current job | SEC API path | Migration note |
|-------------|--------------|----------------|
| Ticker mapping | `/v1/entities/resolve?ticker={ticker}` | Preserve both ticker and CIK in your stored result. |
| CIK mapping | `/v1/entities/resolve?cik={cik}` | Normalize leading zero behavior in tests. |
| Filing search | `/v1/filings` query params | Compare form, filed date, accession number, source URL, and pagination behavior. |
| Filing types and exhibits | [SEC Filing Types and Exhibits](/filing-types-and-exhibits) | Confirm generic retrieval versus workflow-specific normalization. |
| Latest section extraction | `/v1/filings/latest/sections/item_1a?...&mode=compact` | Best for latest filing workflows where compact agent payloads matter. |
| Arbitrary historical URL extraction | Temporary fallback while you qualify historical target support | Keep this path explicit in your dual-run plan. |
| XBRL to JSON | `/v1/statements/{statement_key}` or `/v1/facts` | Compare tags, periods, units, and source filing references. |
| Insider transactions | `/v1/insiders` | Preserve Form 3/4/5 identity and transaction date fields. |
| Compensation | `/v1/compensation?ticker={ticker}` | Validate proxy filing source links and role/year semantics. |

## Request translation examples

### Ticker mapping

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL"
```

### Filing search

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AAPL&form=10-K&limit=10"
```

### Machine-first section extraction

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&mode=compact"
```

## Migration notes

- Pin `secapi-version` in production clients before cutover
- Keep recording `Request-Id` during dual-run testing so request diagnostics stay comparable
- If you depend on arbitrary historical filing URLs for extractor-style requests, keep that path on a temporary fallback until the historical lookup backlog lands
- If you depend on HTML extractor output, treat that as a current gap and stay on the temporary fallback path

## Recommended cutover order

1. issuer resolution
2. filing search
3. statements and facts
4. section extraction in `compact` mode
5. compare and artifact flows once the target plan is enabled

## What to compare during dual-run

| Dimension | What to check |
|-----------|---------------|
| Identity | Ticker, CIK, company name, accession number, form type, filed date, report period |
| Source trail | SEC source URL, filing URL, request ID, freshness, materialization metadata |
| Payload shape | Field names, nullable fields, nested objects, pagination, compact versus verbose response modes |
| Cost and limits | Meter family, expected fanout, retry behavior, and plan fit |
| Agent behavior | Whether the prompt can complete using compact evidence without reading the full filing |
| Fallback policy | Exact workflows that remain on the old provider until parity is accepted |

## Migration acceptance checklist

Before replacing a production sec-api.io workflow, confirm the migrated SEC API path preserves:

- issuer identity: ticker, CIK, and any portfolio identifier you use internally
- filing identity: form type, filing date, accession number, and source URL
- response traceability: request ID and any provenance/freshness fields returned by the endpoint
- fallback policy: what still calls the old path during dual-run testing
- agent contract: the exact compact payload your coding or research agent will receive
