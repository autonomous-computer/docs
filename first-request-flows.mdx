Each flow below has the same goal: retrieve one filing, then preserve its accession number, filing date, form, filing URL, and request ID with any conclusion drawn from it. The latest filing is time-dependent; validate the returned source identity rather than expecting a fixed accession number.

## REST

REST is the direct contract. Use it when your application owns request timeouts, retries, and response handling.

```bash
export SECAPI_API_KEY="secapi_..."
BASE_URL="https://api.secapi.ai"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "$BASE_URL/v1/entities/resolve?ticker=AAPL&view=agent"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "$BASE_URL/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "$BASE_URL/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&view=agent"
```

The first call establishes issuer identity. The second is the cited filing artifact. The section call is evidence for a bounded question, not a replacement for the filing identity. See the [latest filing reference](/api-reference/filings/get-v1-filings-latest) and [section reference](/api-reference/filings/get-v1-filings-latest-sections-section-key).

## JavaScript SDK

The SDK wraps the same public API for application code. It reads `SECAPI_API_KEY` by default.

```ts
import { SecApiClient } from "@secapi/sdk-js"

const client = new SecApiClient()
const entity = await client.resolveEntity({ ticker: "AAPL", view: "agent" })
const filing = await client.latestFiling({ ticker: "AAPL", form: "10-K" })
const section = await client.latestSection({
  ticker: "AAPL",
  form: "10-K",
  sectionKey: "item_1a",
  mode: "compact",
})
```

Save the filing identity from `filing`, then connect a statement or section result to that source only when its returned provenance supports the connection. Installation and language-specific errors are covered in [Libraries and SDKs](/libraries-and-sdks).

## CLI

The CLI is a direct client for terminal use and scripts. It reads `SECAPI_API_KEY` from the environment.

```bash
npm install -g @secapi/cli

secapi entities resolve --ticker AAPL
secapi filings latest --ticker AAPL --form 10-K
secapi sections get --ticker AAPL --form 10-K --section item_1a --view agent
```

Use `secapi <command> --help` to inspect accepted flags and output options in the installed version. See [CLI](/cli) for the command reference.

## Hosted MCP

MCP is for a client that can discover tools and decide which calls to make. Configure the hosted server, then instruct the client to discover the current schema rather than embedding tool names in an application contract.

<Prompt>
Use tools.search to find the issuer-resolution, latest-filing, and filing-section tools. Use tools.describe on each selected tool before calling it. Resolve AAPL, retrieve its latest 10-K, then retrieve Item 1A. Return the company name, accession number, form, filing date, filing URL, requestId, and a short Item 1A summary. Do not make a claim unless it is supported by the retrieved filing or section.
</Prompt>

MCP tool output can include structured content and presentation text. Treat the typed source record and its identifiers as the evidence; preserve any route-provided provenance, freshness, truncation, and trace information. Continue with [MCP workflows](/mcp-workflows).

## Handle failures deliberately

Do not fan out retries. Correct `400` validation errors from the endpoint or tool schema, repair credentials for `401` or `403`, and stop for `402` billing conditions. For `429`, wait for `Retry-After` when present and apply bounded backoff. Keep the API or MCP request identifier with the failed attempt.
