Every first-success path should end with one filing you can identify later. Keep its accession number, form, filing date, filing URL, and request ID when returned. A latest-filing result is time-dependent, so verify the returned filing identity before using it as evidence.

## REST: the reference flow

Use REST when your application owns HTTP, retries, and response parsing.

```bash
export SECAPI_API_KEY="secapi_..."

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

The first call establishes the issuer identity. The second returns the current latest matching filing. Continue with the [latest filing reference](/api-reference/filings/get-v1-filings-latest) when you need the exact parameters and response schema.

To read one section from that current filing, use the documented section key:

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&view=agent"
```

A section answers a narrower question; it does not replace the filing identity you retained from the preceding call.

## JavaScript SDK

Install the SDK, then let it read `SECAPI_API_KEY` from the server-side environment.

```bash
npm install @secapi/sdk-js
```

```ts
import { SecApiClient } from "@secapi/sdk-js"

const sec = new SecApiClient()
const entity = await sec.resolveEntity({ ticker: "AAPL", view: "agent" })
const filing = await sec.agentLatestFiling({ ticker: "AAPL", form: "10-K" })
const section = await sec.agentSection({
  ticker: "AAPL",
  form: "10-K",
  sectionKey: "item_1a",
})

console.log({ entity, filing, section })
```

Expect the same live entity, filing, and section data exposed by the public API. Keep the filing identity from `filing`; inspect each returned object before assuming optional provenance or freshness fields are available. See [JavaScript SDK](/javascript-sdk) for a runnable error-handling example.

## CLI

For a terminal workflow, install the CLI and use the same environment variable:

```bash
npm install -g @secapi/cli

secapi entities resolve --ticker AAPL
secapi filings latest --ticker AAPL --form 10-K
secapi sections get --ticker AAPL --form 10-K --section item_1a --view agent
```

The CLI prints JSON for the current response. Use `secapi <command> --help` for the command version you installed, and continue with the [CLI reference](/cli) for configuration and diagnostic commands.

## Hosted MCP

Install the hosted server in your MCP client, then let the client discover its current tool catalog rather than hard-coding tool names into an application contract.

<Prompt>
Use tools.search to find issuer resolution, latest filing, and filing section tools. Use tools.describe on each selected tool before calling it. Resolve AAPL, retrieve its latest 10-K, then retrieve Item 1A. Return the company name, accession number, form, filing date, filing URL, requestId, and a short Item 1A summary. State when a field is absent instead of inferring it.
</Prompt>

MCP output can include structured content and presentation text. Preserve the typed source record and identifiers; use any returned provenance, freshness, truncation, or trace information only in the context supplied by that tool. [Install MCP](/mcp-install) to configure the connection.

## When the path does not work

Correct `400` validation errors from the route or tool schema. Repair credentials for `401` or `403`, stop for a `402` billing condition, and honor `Retry-After` on `429` when it is present. Preserve the request ID with the failed attempt, then use [request diagnostics](/request-diagnostics).
