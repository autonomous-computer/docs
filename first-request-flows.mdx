<Info>
Choose one investor task, complete it end to end, and keep the identifiers that let someone else reproduce it. A health check proves connectivity; an issuer and filing prove that the integration is useful.
</Info>

## Start in three steps

<Steps>
  <Step title="Resolve one issuer you know">
    Start with a company such as AAPL so you can tell whether the answer looks obviously wrong.
  </Step>
  <Step title="Retrieve a filing with a source trail">
    Ask for the latest 10-K or 10-Q and retain its accession number, form, filing date, and SEC URL.
  </Step>
  <Step title="Add one layer only when you need it">
    Move to statements, section extraction, an SDK, CLI, or MCP after the REST path is clear.
  </Step>
</Steps>

## API first request

```bash
curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"

curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

<Prompt>
Resolve AAPL, retrieve its latest 10-K, and return the issuer CIK plus the filing form, accession number, filing date, source URL, and request ID. Do not summarize the filing until those fields are present.
</Prompt>

## CLI first request

```bash
export SECAPI_API_KEY="your-api-key"
secapi entities resolve --ticker AAPL
secapi filings latest --ticker AAPL --form 10-K
secapi sections get --ticker AAPL --form 10-K --section item_1a
```

## When to use diagnostics

Use [health](https://api.secapi.ai/healthz) and [readiness](https://api.secapi.ai/readyz) endpoints when a normal authenticated request fails or you are operating a service integration. They are useful diagnostic tools, not the first thing a new developer needs to understand.
