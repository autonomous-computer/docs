<Warning>
These are first-party measurements, not an independent study. They describe the named operations, comparands, runner configuration, and dates below. They are not service-level objectives or a guarantee of latency, payload size, cost, or agent quality.
</Warning>

SEC API benchmarks selected SEC-data operations against sec-api.io, financialdatasets.ai, and edgartools. The hosted comparisons were last updated on 2026-03-18. The edgartools comparison is preliminary as of 2026-04-23 and remains subject to final verification.

## Reported comparisons

| Comparison | Operations | Reported p50 range |
| --- | --- | --- |
| sec-api.io | entity resolution, filing search, section extraction, XBRL-to-JSON | SEC API 34-38ms; comparand 231-401ms |
| financialdatasets.ai | income statement, balance sheet, cash flow, metrics | SEC API 55-60ms; comparand 234-375ms |
| edgartools (preliminary) | entity resolution, filing search, section extraction, XBRL-to-JSON | SEC API 34-38ms; comparand 471-3,081ms |

Token estimates use `ceil(payload_bytes / 4)`. They are an approximation for comparing response size, not a model-specific token bill. The edgartools wire-byte measurements include its direct SEC retrieval; that library's zero-key, client-side approach is a valid architectural choice rather than a defect.

## Method

The benchmark uses authenticated production endpoints for hosted APIs and direct SEC EDGAR access for edgartools. It measures wall-clock time from request start to result, including client-side parsing for edgartools. Each operation is run at least five times; p50, p95, and p99 use Hyndman-Fan Type 7 interpolation. The same runner region and machine are used, with concurrent execution.

For edgartools, the persistent cache is cleared between iterations to measure cold fetches. Payload measurement is raw hosted response bytes or the sum of direct SEC HTTP bytes for edgartools. Network conditions, API versions, cache state, rate limits, source availability, request shape, and client behavior all affect a real integration.

## Use and limits

Compare the operations that match your own workload, then run an evaluation with your request mix, region, authentication, payload needs, and acceptable evidence policy. A latency result does not establish filing coverage, analytical accuracy, or source freshness. See [FinanceBench evaluation](/financebench-evaluation) for the separate bounded QA evaluation and [Freshness and source evidence](/freshness-and-trust) for result-level evidence.
