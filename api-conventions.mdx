title: "API conventions"

Use these conventions in every SEC API integration. They make a request reproducible when you need to inspect a result, retry a failure, or ask for help.

## Authenticate requests

Send your API key in `x-api-key`. Do not send an API key as an `Authorization: Bearer` token.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"
```

You can pin a dated API version with `secapi-version`. Requests without that header use the current published version; the response includes the resolved version in `SECAPI-Version`. A pinned version can gain additive fields, so parse the fields you use and allow additional fields in the response.

## Keep the request context

Record these values with any result you rely on:

| Value | Use it for |
| --- | --- |
| `Request-Id` and `requestId` | Identifying one request in logs, troubleshooting, or support |
| `traceparent`, when present | Connecting the request to trace context |
| `accessionNumber` and `filingUrl` | Identifying the SEC filing behind a result |
| Freshness and provenance fields | Understanding the source and currentness of a derived result |
| `SECAPI-Estimated-Cost` and `SECAPI-Token-Count` | Tracking per-call usage |

You can send `x-request-id` with your own correlation ID. Keep the returned `Request-Id` as well.

## Select a response view

Many endpoints support `view=default`, `view=compact`, or `view=agent`.

- `default` returns the full documented response.
- `compact` returns a smaller projection for focused displays or follow-on processing.
- `agent` returns the identifiers, citation pointers, and retrieval fields useful to an agent or downstream model.

View support and projected fields vary by endpoint. Check its reference before relying on a view, and do not treat `compact` and `agent` as interchangeable.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

## Page through list results

When a response includes `hasMore` and `nextCursor`, send `nextCursor` as the next request's `cursor`. Stop when `hasMore` is `false`.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AAPL&limit=25"
```

Some list-shaped responses return the complete set without cursor fields. Use the endpoint reference for its supported `limit` values and pagination behavior.

## Handle errors deliberately

Error responses include a machine-readable `code`, a message, and a request ID. Read the response before retrying.

| Status | What to do |
| --- | --- |
| `400` | Correct the path, query, or JSON body using the endpoint reference. |
| `401` | Confirm that a valid key is sent in `x-api-key`; use the signed-in dashboard for browser-only actions. |
| `402` | Resolve the billing or entitlement condition in the response before retrying. |
| `429` | Wait for `Retry-After` when present; otherwise use bounded exponential backoff and do not send concurrent retries. |
| `5xx` | Retry once with bounded backoff, then retain the request ID. |

Read [Troubleshooting](/troubleshooting) for error-specific guidance. Read [SDK reliability](/sdk-reliability) before adding retries around an SDK.

## Read statement periods correctly

Statement periods include `periodBasis`. Balance-sheet values are point-in-time. Quarterly income-statement and cash-flow values can be year-to-date as filed. Check `periodBasis` before comparing a quarterly value with a discrete quarter.

## Choose REST or MCP

Use REST or an SDK when your application needs direct control over the HTTP request and response. Use hosted MCP when an MCP-aware client needs SEC API tools in a research workflow. MCP uses JSON-RPC and has its own quota and error envelopes.

Start with [your first request](/getting-started), then read [MCP installation](/mcp-install) when you are connecting an MCP client.
