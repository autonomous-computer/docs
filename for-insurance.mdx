Use this guide to assemble a repeatable renewal or counterparty review packet for a public company. Preserve the filings and event records that support the packet; your team retains responsibility for underwriting criteria, scoring, and coverage decisions.

<Prompt>
Build a D&O renewal research note for [TICKER] as of [DATE]. Resolve the issuer, retrieve its latest 10-K and relevant 8-Ks, then read Item 1A risk factors and Item 7 MD&A. Add current named-executive compensation, recent Form 4 activity, and the latest available institutional-holder context. Use `owners.institutional_holders` with the issuer ticker or issuer CIK; the manager-specific 13F holdings tool requires a named manager CIK. Separate facts from underwriting interpretation. Cite each finding with form, filing date, accession number, and section or transaction reference. Flag only disclosed auditor changes, restatement language, control weaknesses, executive departures, or unusually concentrated ownership; do not infer fraud, claim likelihood, or coverage. Use entities.resolve, filings.search, sections.get, comp.list, insiders.list, and owners.institutional_holders.
</Prompt>

## Build the review packet

| Review question | Route | What to retain |
| --- | --- | --- |
| Which issuer is this? | `GET /v1/entities/resolve` | Resolved name, ticker, CIK, and the identifier used downstream. |
| What did it file? | `GET /v1/filings` | Form, filing date, accession number, and filing URL. |
| What did the filing say? | `GET /v1/filings/latest/sections/{section_key}` | Filing identity, section key, text, and any route warnings. |
| Is there a disclosed event to review? | `GET /v1/events/restatements`, `GET /v1/events/auditor-changes`, or `GET /v1/events/enforcement` | Returned event record, source link or filing identity, and response diagnostics. |

Use each route for its documented record type. An event result is a research input, not a determination of fraud, claim likelihood, or coverage.

## Check one renewal candidate

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AIG"

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings?ticker=AIG&forms=10-K,8-K&limit=10"

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/events/auditor-changes?ticker=AIG&date_from=2024-01-01&limit=10"
```

For a filed controls disclosure, retrieve the source section separately:

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_9a?ticker=AIG&form=10-K"
```

## Keep the review boundary clear

Capture accession numbers, filing URLs, section keys, dates, and `requestId` with the packet. Treat missing, partial, or unavailable results as review conditions, not as evidence that a condition does not exist. The API does not replace primary-source review, legal analysis, underwriting judgment, or your organization’s retention policy.
