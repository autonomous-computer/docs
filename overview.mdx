Start with a real issuer and a narrow question. SEC API supports filing search, sections, structured statements, ownership data, market utilities, and agent tooling; you do not need to learn every endpoint before shipping a useful result.

## Start with an issuer workflow

For most new integrations, use this order:

1. [Resolve an entity](/api-reference/entities/get-v1-entities-resolve) by ticker, CIK, FIGI, or name.
2. Retrieve the issuer's [latest filing](/api-reference/filings/get-v1-filings-latest).
3. Read a filing section or request the [all-statements bundle](/api-reference/statements/get-v1-statements-all).
4. Keep the filing accession number and request metadata with your result.

This path confirms that authentication, identifiers, filing access, and the response shape work before you add search, pagination, or background jobs.

## Choose the surface, not a different data product

REST is the underlying HTTPS API. The SDKs and CLI package common requests for their native environments. Hosted MCP exposes tools for MCP clients. Choose one surface for your first workflow; the documented REST endpoint remains the best place to verify exact parameters and response fields.

| Surface | Use it when | What to read |
| --- | --- | --- |
| REST | You control HTTP requests or use another language | [API overview](/api-overview) |
| SDK | You use JavaScript, Python, Go, or Rust | [Libraries and SDKs](/libraries-and-sdks) |
| CLI | You work in a terminal or need a simple smoke check | [CLI](/cli) |
| MCP | Your client can call hosted MCP tools | [MCP workflows](/mcp-workflows) |

## Keep the evidence that explains a result

For filing-backed output, save the accession number, filing date, and source URL when present. Also retain `requestId`; response-specific `provenance`, `freshness`, `materialization`, and trace fields provide additional context when the endpoint returns them. Not every route or compact view includes every metadata field.
