Start with one issuer, one request, and one result you can trace back to a filing. You do not need to learn the whole API to ship a useful workflow.

## Get to a real result

The fastest path is:

1. [Create an API key and resolve an issuer](/getting-started).
2. Retrieve that issuer's [latest matching filing](/api-reference/filings/get-v1-filings-latest).
3. Keep the filing identity with anything you derive from it.

For example, a latest 10-K response can give you the filing's accession number, date, URL, and request ID. The latest filing changes as new filings arrive, so read those returned values instead of expecting a fixed accession number.

## Choose your interface after the request works

REST is the public HTTPS contract. The SDKs, CLI, and hosted MCP give you other ways to reach documented API capabilities.

| Interface | Start here when | Next action |
| --- | --- | --- |
| REST | You control HTTP and need the exact request and response contract. | [First request flows](/first-request-flows) |
| SDK | You are writing JavaScript, Python, Go, or Rust. | [Libraries and SDKs](/libraries-and-sdks) |
| CLI | You want a terminal check or shell automation. | [CLI](/cli) |
| MCP | Your MCP client should discover SEC API tools. | [Install MCP](/mcp-install) |

## Keep enough context to explain the result

For filing-backed output, retain the returned accession number, form, filing date, filing URL, and `requestId` when they are present. Those fields connect your result to a specific filing and API request. Some routes and response views also return provenance, freshness, materialization, warning, or trace fields. Treat them as route-specific context, not fields guaranteed on every response.
