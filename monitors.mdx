title: "Monitor SEC Filing Events"
description: "Create saved SEC filing searches, review matches with an API key, and deliver scheduled alerts to email or signed webhooks."

Monitors are saved searches that alert your organization when a new SEC filing matches a query. Use them for material 8-Ks, 13F holdings changes, insider activity, auditor changes, and filing-language alerts.

## Set up a monitor

Create, edit, pause, and delete monitors from the signed-in [SEC API dashboard](https://secapi.ai/app/webhooks). These are organization control-plane actions: they require a WorkOS browser session and do not accept an API key. The same boundary applies to webhook endpoint setup, secret rotation, test deliveries, and stream-subscription changes.

For each monitor, choose a clear name, keyword query, optional company/form/date filters, and either an email destination or an organization webhook endpoint subscribed to `monitor.match`.

## Read monitor data with an API key

After setup, these read operations accept `x-api-key` and are suitable for agents, scheduled checks, and audit reports:

```bash
# List active monitors for your organization.
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/monitors"

# Preview the filings currently matched by one monitor.
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/monitors/mon_123/matches?limit=10"
```

`GET /v1/monitors/{monitor_id}/matches` runs the saved search on demand. It is useful for validating a query and investigating an alert, but it does not trigger delivery.

## Delivery choices

### Email

Choose email when the alert belongs in an inbox. Each match batch produces one transactional email with HTML and plaintext content plus a one-click unsubscribe path. A destination can be `active`, `unsubscribed`, `bounced`, `complaint_suppressed`, or `suppressed`; non-active destinations do not receive new alerts.

### Webhook

Use an organization webhook endpoint when a service should receive signed `monitor.match` events. In the dashboard, add a public HTTPS destination and subscribe it to `monitor.match`; then create the monitors that should feed it. Save the signing secret when the dashboard shows it, and verify the exact raw request body before parsing JSON. See [Build a Filing Monitor](/tutorials/build-filing-monitor) for a complete handler.

## Timing, retries, and limits

SEC API checks active monitors on a 15-minute cadence. Typical filing-to-alert latency is about half a sweep; worst-case latency is one full sweep plus delivery time. Each dispatch emits `monitor.match` to subscribed organization webhook endpoints.

Transient email failures are retried automatically. To avoid alert storms, the default limits are 100 sends per monitor per hour and 1,000 sends per organization per day. Narrow a query or split distinct alert jobs into separate monitors when those limits are not a good fit.
