---
title: "Troubleshooting"
description: "Resolve SEC API authentication, billing, rate-limit, and request failures from the HTTP status and error code"
---

Start with the failed response. Record its HTTP status, error `code`, `Request-Id`, and `traceparent` when present. Then reproduce the request once with safe, redacted inputs.

```bash
curl -i -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"
```

If this request fails, resolve the credential or organization condition before debugging a filing, statement, SDK, or MCP workflow. Machine requests use `x-api-key`; browser account actions use a signed-in browser session.

## Use the HTTP status and code together

| Status or code | Meaning | First action |
| --- | --- | --- |
| `400`, `invalid_query_parameter`, or `validation_error` | The path, query, or body does not match the route contract. | Compare the request with the endpoint reference and remove unsupported fields. |
| `401 missing_api_key` | No API key reached the API. | Send `x-api-key` from the process making the request. |
| `401 authentication_failed` | The credential is invalid, revoked, or belongs to another organization. | Replace the key through the appropriate signed-in flow. |
| `401 missing_browser_session` | A browser-only action was called as a machine request. | Complete it in the signed-in dashboard. |
| `402` or `billing_required` | Billing, plan, grant, or budget state blocks the request. | Inspect billing and limits with the same credential; resolve the stated condition before retrying. |
| `403` | A valid credential lacks authorization for the resource or feature. | Check the credential's access and the plan feature. |
| `429` or `rate_limit_exceeded` | A rate or quota window applies. | Honor `Retry-After`, reduce fan-out, and retry only after the indicated interval. |
| `5xx` | The request could not complete. | Retry once with bounded backoff, then retain the request context. |

Read the [error code catalog](/error-code-catalog) for code-specific guidance.

## Resolve billing and quota stops

```bash
curl -sS -H "x-api-key: $SECAPI_API_KEY" "https://api.secapi.ai/v1/billing"

curl -sS -H "x-api-key: $SECAPI_API_KEY" "https://api.secapi.ai/v1/limits"
```

`billing_budget_approval_required` means a configured approval threshold stopped the request. `billing_budget_hard_cap_exceeded` means the configured hard cap stopped it. `ai_query_quota_exceeded` means the applicable AI-query allowance is exhausted. Stop rather than repeatedly retrying these outcomes. See [Agent billing](/agent-billing), [GET /v1/limits](/api-reference/platform/get-v1-limits), [API keys, usage, and billing](/auth-and-pricing), and [Plans and pricing](/plans-and-pricing).

## Diagnose a partial data result

A partial or degraded result is not necessarily a transport failure. Inspect the returned capability, warning, provenance, period, freshness, or status-note fields for that route. Preserve usable source evidence and make the limitation visible to your caller; do not silently substitute a different issuer, period, or data surface.

## Diagnose hosted MCP calls

Confirm that the client connects to `https://api.secapi.ai/mcp`, sends the configured API key when using API-key authentication, and has refreshed its tool list. JSON-RPC `-32004` indicates a tool execution-budget stop, `-32005` an AI-query or MCP quota stop, and `-32006` a tool-level protection stop. Narrow the request or wait for the applicable limit instead of retrying blindly.

For support, include the method or tool name, redacted request, HTTP status, error code, `Request-Id`, trace context, and the outcome of `GET /v1/limits` with the same credential. Never include an API key.
