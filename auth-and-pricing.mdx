## Browser dashboard sessions

Dashboard login uses WorkOS to create an `HttpOnly`, `SameSite=Lax` browser-session cookie named `omni_datastream_session`. Google, GitHub, and email-based WorkOS sign-in are supported by the WorkOS tenant configuration. Public signup is controlled by the `SECAPI_AUTH_PUBLIC_SIGNUP_ENABLED` flag.

The cookie is an AES-256-GCM sealed payload that also carries the WorkOS refresh token, so the dashboard stays signed in without re-prompting: the short-lived WorkOS access token is refreshed server-side before it expires. Sessions use a 30-day sliding idle window (re-issued on each refresh) with a 90-day absolute cap measured from first login.

`GET /auth/session` returns `expiresAt` and `refreshAvailable: false` — refresh is handled server-side via the sealed cookie, not by the client. The dashboard sends users back through `/login` when the idle window lapses, the 90-day absolute cap is reached, or the underlying WorkOS token can no longer be refreshed or verified.

Dashboard control-plane routes require a human WorkOS browser session with the right permissions. API-key-authenticated callers cannot create or revoke dashboard API keys, mutate billing state, update profile/settings, or submit account deletion requests.

## Agent-native entry path

- start at the public `/agents` overview when the caller is an external coding agent or operator
- mint a short-lived sponsor token with `POST /v1/agent/bootstrap_tokens` from a human-authenticated org session
- exchange that sponsor token at `POST /v1/agent/bootstrap`
- preserve the returned API key secret once, then treat it as the machine credential for REST, SDK, CLI, or hosted MCP

## API key revocation SLA

Revoking a key through the dashboard or `DELETE /v1/api_keys/:keyId` marks that key `revoked` in Postgres and, when Redis is available, publishes a short-lived revocation marker for API replicas.

- the replica that processes the revoke stops serving the key immediately
- with Redis healthy, other API replicas reject cached copies within `5` seconds
- if Redis is unavailable but Postgres is healthy, cached API key principals expire and re-check Postgres within `60` seconds
- if Redis and Postgres are both unavailable after a key has already been cached on a replica, stale-while-revalidate is capped at `6m5s`

For suspected key theft, revoke the key, create a replacement, rotate callers, and monitor for `401` responses and unexpected usage. Contact support for a forced fleet cache flush if you need emergency containment beyond the normal propagation window.

## Default onboarding motion

<Steps>
  <Step title="Create your org and API key">
    Create a free account, then create your first org-scoped API key explicitly from the dashboard or account API using a WorkOS browser session, where it is revealed exactly once.
  </Step>
  <Step title="Receive the starter grant">
    Every new org starts with `250` included calls; API keys are created explicitly and revealed once.
  </Step>
  <Step title="Make one real request">
    Resolve an issuer, fetch a filing, or run one statement workflow before you think about plan changes.
  </Step>
  <Step title="Attach billing only when the work is real">
    After the starter grant is exhausted, activate Pay As You Go or move directly to a commit plan.
  </Step>
</Steps>

- redistribution, embedding, resale, and broader commercial rights begin at `Commercial`

## WebSocket rate posture

Chat messages sent through the SEC API Agent WebSocket runtime are rate-limited per user at a configurable requests-per-minute ceiling (default: 30 RPM). Non-chat messages — permission responses, plan approvals, label and source operations — are exempt because they are responses to server-initiated prompts, not user-initiated load.

Rate-limited requests receive a clear error message with a retry-after interval. The WebSocket connection stays open. This posture protects shared infrastructure without punishing interactive workflows.

- `RATE_LIMIT_CHAT_RPM`: per-user ceiling (default 30)
- `RATE_LIMIT_ENABLED`: feature flag (default true in production)

## Dashboard surfaces

The dashboard mirrors the account and billing API state instead of inventing client-side entitlements.

- Overview: principal, org, plan, quota, recent usage, and key context
- API Keys: active-key count, create/reveal-once, copy, revoke, and revocation status
- Usage: time series, endpoint breakdown, request activity, and export controls
- Billing and Credits: subscription, PAYG, budget, quota, starter grant, and billing portal entry points
- Settings: profile, organization, appearance, security, logout, and deletion-request controls

Billing mutations are server-authoritative. The browser only receives Stripe Checkout or Billing Portal URLs; it does not decide plan state locally.

These endpoints back the hosted MCP surface. They are not part of the primary human-auth flow — API keys cover every REST, SDK, and CLI client.

- protected resource metadata: `https://api.secapi.ai/.well-known/oauth-protected-resource`
- authorization server metadata: `https://api.secapi.ai/.well-known/oauth-authorization-server`
