---
title: "API keys, usage, and billing"
description: "Authenticate machine requests with x-api-key, inspect account-specific limits, and keep dashboard actions separate"
---

Use an API key for machine access to REST, SDK, CLI, and server-to-server MCP requests. Create, reveal, rotate, and revoke keys in the signed-in [SEC API dashboard](https://secapi.ai/app).

## Send the credential correctly

```bash
export SECAPI_API_KEY="secapi_..."

curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/limits"
```

The key belongs in `x-api-key`. Do not send it as `Authorization: Bearer`, add it to a URL or request body, put it in browser-delivered code, or commit it to source control. A browser session is for signed-in dashboard actions, not a substitute for an API key on a data request.

## Use the credential that matches the job

| Job | Credential and location |
| --- | --- |
| Read SEC data from code, an SDK, CLI, or server-to-server MCP client | API key in `x-api-key`, held by your server, worker, shell, or MCP client |
| Create, reveal, rotate, or revoke an API key | Signed-in dashboard session |
| Change billing, payment methods, or budgets | Signed-in dashboard session with the required access |

## Inspect the account that will run the workload

After the first request succeeds, inspect the authenticated organization's current state with the same API key:

```bash
curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/billing"

curl --fail-with-body -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/billing/rates"
```

Expect organization-specific usage, plan, billing, or rate context where the endpoint exposes it. Limits, rates, and entitlements depend on the authenticated organization and endpoint. Use the response and [plans and pricing](/plans-and-pricing), rather than a copied example, to plan sustained traffic.

## Keep the response context that matters

For requests your users depend on, record `Request-Id` and `traceparent` when present. Also retain `SECAPI-Meter-Class`, `SECAPI-Estimated-Cost`, `SECAPI-Token-Count`, and `Retry-After` when supplied. They describe one response; they do not replace checking the account's current billing and limits.

## Rotate without interrupting callers

1. Create a replacement key in the dashboard.
2. Update the server-side secret where the application runs.
3. Verify the replacement with `GET /v1/limits`.
4. Revoke the old key after callers are using the replacement.

For a suspected leak, revoke the exposed key before creating and deploying its replacement. Continue with [API conventions](/api-conventions) for response behavior or [request diagnostics](/request-diagnostics) for a failed request.
