# API Overview

## What the API is optimized for

- grounded investor workflows
- compact payloads that work well in agents
- explicit provenance and freshness
- predictable wire contracts across REST, SDKs, CLI, and hosted MCP

## Core design rules

- version requests with `Omni-Version`
- correlate requests with `Request-Id`
- use cursor pagination on list endpoints
- expose provenance, freshness, and materialization metadata on public market intelligence paths
- prefer structured contracts over raw passthroughs

## What makes the contract different

- `provenance`: where the data came from
- `freshness`: how current the data is
- `materialization`: how the current output was produced

These fields matter because investor agents need more than a value. They need the trail behind the value.

## Event delivery rules

- webhook delivery uses signed `POST` requests with `x-omni-signature`
- delivery headers include `x-omni-event-id`, `x-omni-event-type`, and `x-request-id`
- current event families include artifacts, webhooks, and stream subscriptions

## Product posture

Datastream is not trying to be a generic raw filing tunnel. The product is designed around the workflows serious investors and investor agents actually repeat.
