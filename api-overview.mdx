<Info>
SEC API gives you the same filing, statements, ownership, and search primitives through several interfaces. Start with REST when you need to inspect a response. Use an SDK or CLI when you are building a repeatable workflow. Use hosted MCP when an MCP-aware agent should call the data layer directly.
</Info>

## Authentication

API keys are the primary machine-auth method for all plans. Use the `x-api-key` header on every REST request; the same credential works across SDK, CLI, and server-to-server MCP clients. See [Auth, Billing, and First Utility](/auth-and-pricing) for the full model. WorkOS powers dashboard login and remains the enterprise SSO boundary as organization-level access expands.

## Core design rules

<Steps>
  <Step title="Identify the source record">
    Keep the issuer identifier, filing accession number, filing date, and source URL with any conclusion drawn from a filing.
  </Step>
  <Step title="Correlate the request">
    Keep `Request-Id` in logs and support requests. It identifies the API call without exposing your key.
  </Step>
  <Step title="Use the documented response shape">
    Start with the default response when you need the surrounding context. Use `view=agent` only when the endpoint documents an agent projection.
  </Step>
  <Step title="Handle partial state deliberately">
    A documented `degraded` or `unavailable` state is information about the response, not a value to smooth over in downstream code.
  </Step>
  <Step title="Page when the endpoint pages">
    List endpoints that expose `nextCursor` and `hasMore` use cursor pagination. Do not assume every list-shaped response does.
  </Step>
  <Step title="Pin production clients">
    Send a documented `secapi-version` when you are ready to pin a production integration to a dated contract.
  </Step>
</Steps>

## What makes the contract different

- `provenance`: where the data came from
- `freshness`: how current the data is
- `materialization`: how the current output was produced
- `trace`: how to get from a filing-derived value back to the source filing and page when available

These fields matter because investor agents need more than a value. They need the trail behind the value.

## Event delivery rules

- webhook delivery uses signed `POST` requests with `x-secapi-signature`
- delivery headers include `x-secapi-event-id`, `x-secapi-event-type`, and `x-request-id`
- current event families include artifacts, webhooks, and stream subscriptions

<Warning>
SEC API is not trying to be a generic raw filing tunnel. It is designed around the workflows serious investors and investor agents actually repeat.
</Warning>
