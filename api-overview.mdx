<Info>
Start with REST when you need to see the exact request and response. Move to an SDK or CLI when the call belongs in a repeatable workflow. Use hosted MCP when an MCP-aware client should call SEC API tools directly.
</Info>

## Authentication

Use an API key in `x-api-key` for REST. The SDKs and CLI read the same API-key credential from their client configuration or environment. Keep dashboard sign-in and browser-only account actions separate from machine requests. See [API keys, usage, and billing](/auth-and-pricing) for the credential boundary.

## Build against the contract

<Steps>
  <Step title="Prove one request first">
    Follow the <a href="/getting-started">first request</a> before introducing an SDK, an agent, or concurrent traffic.
  </Step>
  <Step title="Use the documented parameters">
    The API reference is authoritative for accepted identifiers, filters, response views, and limits. Do not infer them from a similar endpoint.
  </Step>
  <Step title="Keep the identifiers that matter">
    Retain `Request-Id` for support and, for filing-derived work, the issuer identifier, accession number, filing date, and filing URL.
  </Step>
  <Step title="Choose a response view deliberately">
    Start with the default response when you need context. Use `view=agent` or `view=compact` only where the endpoint documents that projection.
  </Step>
  <Step title="Respect pagination and rate limits">
    Endpoints that return `nextCursor` and `hasMore` use cursor pagination. For `429`, wait for `Retry-After`; endpoint-specific limits belong in the reference.
  </Step>
  <Step title="Pin a production integration">
    Send a documented `secapi-version` when you are ready to pin to a dated contract.
  </Step>
</Steps>

## Response metadata

- `Request-Id` identifies one API request for logs and support.
- Filing identifiers such as `accessionNumber` and `filingUrl` identify the source filing when present.
- Some endpoints expose provenance, freshness, materialization, or trace fields. Their availability and shape are endpoint-specific.

Preserve the fields your endpoint returns when they support a user-visible result.
