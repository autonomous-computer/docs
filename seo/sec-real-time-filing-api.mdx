Real-time filing delivery tells a worker that new matching filing work is available. It does not itself validate the filing, guarantee an investor conclusion, or replace retrieving the cited filing record.

## Pick a delivery model

| Need | Mechanism | Operational responsibility |
| --- | --- | --- |
| Push a matching saved-monitor event to a public service | dashboard-configured webhook | verify signature, deduplicate, queue work |
| Process events with explicit checkpoints | stream-subscription polling | commit cursor only after durable processing |
| Maintain a live connection without exposing an API key in a URL | subscription-backed, ticketed WebSocket | mint short-lived ticket and reconnect deliberately |
| Simple periodic check | `GET /v1/filings/latest` | choose form and own polling cadence |

Create saved monitors, webhook endpoints, subscription changes, secret rotation, tests, and replay policy in the signed-in [dashboard](https://secapi.ai/app/webhooks). Those organization changes are human-authenticated; an API key is not the setup credential.

## Poll safely with a cursor

```bash
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/stream_subscriptions/str_123/events?limit=25"
```

The expected response is an ordered `data` batch with `nextCursor`, `replayCursor`, `requestId`, and `traceparent`. Store each event id in the same durable transaction that creates downstream work. Advance the checkpoint only after that write succeeds. Cursor order is recorded event order, not an SEC filing timestamp.

## Use a ticketed WebSocket

```bash
curl --fail-with-body -X POST \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "content-type: application/json" \
  -d '{}' "https://api.secapi.ai/v1/stream/tickets"

npx wscat -c "wss://api.secapi.ai/v1/stream/ws?stream_id=$SECAPI_STREAM_ID&ticket=$SECAPI_STREAM_TICKET"
```

The `stream_id` must identify an active WebSocket stream subscription. Mint a short-lived ticket before connecting so a long-lived API key does not enter a browser-visible WebSocket URL. The JavaScript SDK exposes a filing-stream WebSocket helper that requires this subscription-backed stream ID; use the REST ticket contract when you need to control token handoff yourself.

## Process a monitor webhook as evidence delivery

For `monitor.match`, verify `x-secapi-signature` against the exact raw body before parsing JSON; reject stale timestamps, deduplicate delivery/event IDs, queue work, and return a quick `2xx`. Retain the monitor ID, event ID, request ID, accession number, and filing URL. Then retrieve the filing or named section before summarizing it.

## Timing and failure boundaries

- Delivery is not a universal SEC freshness guarantee. Inspect returned filing identifiers and source timestamps for the event you received.
- Webhooks can be retried and event batches can be replayed. Consumers must be idempotent.
- Do not advance a cursor after a partial failure or derive a checkpoint from an array position, wall-clock time, or accession number.
- A latest-filings poll without `form` means the latest filing of any type; pin a form family for a 10-K, 8-K, or other focused workflow.
