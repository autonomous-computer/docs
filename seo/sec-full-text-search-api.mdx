---
title: "SEC Full-Text Search API"
description: "Search exact terms across SEC filing records and extracted sections, then inspect the source filing before using a result."
---

Use full-text search when a phrase, policy name, or contract term matters. It searches both filing records and extracted sections, so the response has two lists to review.

## Search filing content

```bash
curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/search/fulltext?q=material%20weakness&ticker=AAPL&form=10-K&limit=5"
```

`q` is required. Constrain the query with a ticker or CIK, form, accession number, filing year, `fy`, or `date_from` and `date_to`.

```json
{
  "object": "fulltext_search",
  "query": "material weakness",
  "filings": { "object": "list", "data": [], "count": 0 },
  "sections": {
    "object": "list",
    "data": [],
    "count": 0,
    "degradedState": null
  },
  "requestId": "req_example"
}
```

The example intentionally shows an empty result: matched fields depend on the filing corpus at request time. In a real response, retain filing identity and source URLs, then open the filing or retrieve its section before drawing a conclusion.

## When to use another path

- Use [semantic search](/seo/sec-semantic-search-api) for related concepts expressed in different language.
- Use [section search](/api-reference/sections/get-v1-sections-search) when you need a cited extracted passage.
- Use [filing search](/seo/sec-filing-search-api) when the question is which filings exist, not what they say.

## Production notes

An empty response is not evidence that the issuer never made a disclosure. Exact wording, form selection, date filters, and search-backend availability all affect the result. Log `requestId` and retry only bounded, retryable server failures.
