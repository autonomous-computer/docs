# Support and admin workflows

## Admin endpoints

- `GET /v1/admin/orgs`
- `GET /v1/admin/orgs/{org_id}`
- `GET /v1/admin/orgs/{org_id}/requests/{request_id}`
- `GET /v1/admin/orgs/{org_id}/deliveries/summary`
- `GET /v1/delivery/events/export`
- `GET /v1/observability/export`

## Required scope

- operator workflows require an API key with `admin:operator`
- wildcard bootstrap keys with `*` also work for operator flows
- prefer `SECAPI_OPERATOR_API_KEY` in support scripts; `SECAPI_OPERATOR_API_KEY` remains accepted by smoke and release automation

## Recommended support sequence

1. capture `Request-Id`
2. fetch org summary
3. fetch request diagnostics
4. fetch delivery summary
5. export event and observability context

## Automation

- `bun run ops:support-bundle -- --org-id <org> --request-id <request>`
- `bun run ops:alert-matrix`

## Verified production support posture

- `GET /v1/observability` and `GET /v1/observability/export` are live and operator-only
- `bun run ops:launch-readiness` returns `status: ready` against production
- support bundles now work with operator-scoped keys and wildcard bootstrap keys
- freshness, company-mapping, and external-reference validation artifacts are published separately and should be checked alongside support readiness when a workflow looks suspicious

## Output expectations

- support bundles land in `tmp/support-bundles/`
- alert mapping lands in `ops/alert-matrix/latest.json`
- workflow escalations create Linear issues with runbook metadata
