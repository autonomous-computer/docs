Use hosted MCP when the client needs to choose an SEC API capability at runtime. The first useful workflow is deliberately small: discover the tools, retrieve one filing and one section, then summarize only the evidence returned. [Install SEC API MCP](/mcp-install) before running this job, then review [pricing and limits](/evaluate/pricing-and-limits) before expanding it into repeated work.

## Ask for one bounded result

Give a connected MCP client this prompt:

<Prompt>
Use tools.search to find issuer resolution, latest-filing, and filing-section tools. Use tools.describe before each selected call. Find NVIDIA's latest 10-K and retrieve Item 1A. Return company name, accession number, form, filing date, filing URL, section title, requestId, and a five-bullet summary. Separate retrieved facts from interpretation and say when the filing or section does not support an answer.
</Prompt>

A successful run returns a filing identity and a bounded section result. A later run can legitimately select a newer accession number. An uncited summary is not a substitute for the filing artifact.

## Let the server supply the tool contract

`tools.search` accepts a natural-language job. `tools.describe` returns the current input schema for an exact selected name. `tools/list` is available for clients that need a catalog and supports `core` or namespace filters.

Do not freeze a copied tool list into a durable agent instruction. Tool names, arguments, annotations, and availability belong to the live server contract. Keep `structuredContent` as the typed result; text content can assist the model but should not replace the returned data.

## Retain the review record

When available, retain:

- issuer identity;
- `accessionNumber`, `form`, `filingDate`, and `filingUrl`;
- `requestId` and `traceparent`; and
- provenance, freshness, pagination, truncation, and capability fields.

Search and section retrieval identify evidence within a defined scope. A match, a rank, or an extraction does not by itself establish a fact beyond that record.

## Handle failure at the right layer

`GET /mcp` is public discovery metadata. Authenticated JSON-RPC calls go to `POST /mcp` with `x-api-key`. Authentication, generic rate-limit, billing, and entitlement failures can occur before JSON-RPC dispatch; dispatched tool, quota, timeout, and resource failures use JSON-RPC error envelopes. Preserve `error.data.requestId` and `error.data.traceparent` when present.

Do not retry invalid arguments, missing credentials, billing refusals, or quota exhaustion blindly. Check `Retry-After` for rate limits and correct the cause before another call. Use REST or an SDK when your deployed application needs to own stable request loops, timeout policy, and typed application errors.
