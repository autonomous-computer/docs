MCP is useful when a research client can choose tools at runtime. The operating sequence is simple: discover a tool, inspect its schema, retrieve one source record, then summarize only that record. [Install SEC API MCP](/mcp-install) first.

## A complete first job

Give the connected client this prompt:

<Prompt>
Use tools.search to find tools for issuer resolution, latest filing, and filing sections. Use tools.describe before every selected call. Find NVIDIA's latest 10-K and retrieve Item 1A. Return company name, accession number, form, filing date, filing URL, section title, requestId, and a five-bullet summary. Separate retrieved facts from interpretation. Do not infer facts that are not in the filing or section.
</Prompt>

A successful job returns a filing identity and a bounded section result. A newer filing on a later run may legitimately have a different accession number; an uncited summary is not a replacement for the source artifact.

## Use discovery as the contract

The server owns its current tool names, schemas, annotations, and limits. Start with `tools.search` using a natural-language job, then use `tools.describe` with the exact selected name. `tools/list` is available when a client needs the catalog and can be filtered with `core` or a namespace. Do not hard-code a copied tool catalog into a durable agent workflow.

`structuredContent` is the canonical typed tool payload. Text content may be useful to the model, but preserve the typed output and returned fields that identify the source.

## Produce a reviewable result

Persist these fields whenever they are returned:

- issuer identity
- `accessionNumber`, `form`, `filingDate`, and `filingUrl`
- `requestId` and `traceparent`
- provenance, freshness, pagination, truncation, and capability fields

Treat missing evidence as missing. A section can support a claim about the section's text; it does not automatically establish facts outside that scope.

## Move from exploration to production

Use MCP for interactive research and agent-led discovery. Use REST or an SDK when your application needs a stable request loop, typed application errors, deployment-controlled timeouts, or its own retry policy. In either case, keep work bounded, inspect billing before repeated work, and avoid retrying authentication, validation, billing, or quota failures without a corrective action.
