Use SEC API MCP for a defined research job, such as locating a filing, reading a section, or checking reported ownership. Give the client a narrow question and require the filing identifiers that support its answer.

Install the server before starting: [Install SEC API MCP](/mcp-install).

Need an API key? [Create an SEC API account](https://secapi.ai/signup) before connecting your client.

## Start with one cited filing

Use a prompt that names the issuer, filing type, and evidence you expect back:

<Prompt>
Find NVIDIA's latest 10-K and summarize Item 1A. Return the accession number, filing date, filing URL, request ID, and any source or freshness fields included in the result.
</Prompt>

This establishes a useful boundary: the client may summarize the filing, but the response must still identify the SEC document behind the summary.

## Find the right tool

Use the server's tool catalog as the source of truth. Start with `tools.search` when you know the job but not the tool name, then use `tools.describe` before a call that needs exact arguments.

| Research job                            | Search terms to use                    |
| --------------------------------------- | -------------------------------------- |
| Identify an issuer from a ticker or CIK | `company identity`                     |
| Find a filing by company, form, or date | `filing search`                        |
| Retrieve a section such as Item 1A      | `filing section`                       |
| Read reported financial values          | `financial statements` or `XBRL facts` |
| Review institutional ownership          | `institutional holders`                |

Tool names, inputs, and annotations can change as the server evolves. Keep discovery in the workflow rather than hard-coding a catalog from this page.

## Preserve the evidence

For every filing-based answer, keep these fields when they are returned:

- `accessionNumber`, `filingDate`, and `filingUrl`
- `requestId` and `traceparent`
- source URLs, citations, provenance, and freshness fields
- pagination, truncation, or response-limit signals for list results

The identifiers let you trace a conclusion to a filing and diagnose a failed or partial request later.

## Bound the request

Specify the company, form type, date range, section, and output fields that the task needs. For a list result, set a small limit first, inspect the result, then widen the query only when needed. Use the documented response controls when you want a compact text representation; structured MCP content remains the canonical payload.

Do not automatically replay a call after an authentication, billing, or quota response. For a timeout, narrow the request before trying again and retain the request ID from the failed response.

## Move from research to code

Use MCP when the task belongs in an MCP-aware client. Use an SDK or REST when you need a stable application integration, typed client behavior, or direct control over the request and response.
