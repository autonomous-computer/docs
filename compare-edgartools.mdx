[edgartools](https://github.com/dgunning/edgartools) is a Python library for working directly with SEC filings. It also provides LLM-oriented workflows, a local MCP server, and local-storage options for downloaded SEC data. SEC API is a hosted, managed data service with REST, SDK, CLI, and hosted MCP interfaces. Neither is a universal replacement for the other: choose based on where parsing, operations, caching, and source-record handling belong in your application.

## Start with the operating model

| Your requirement | A reasonable starting point |
| --- | --- |
| Inspect or customize document parsing in a Python research workflow | edgartools and direct SEC source documents |
| Run an LLM workflow or MCP server beside local Python data and parsing | edgartools |
| Cache downloaded SEC data under your own local-storage policy | edgartools local storage |
| Call a managed, documented HTTP contract from a service, worker, or non-Python runtime | SEC API REST or an SDK |
| Give an MCP client a hosted tool surface without operating a local MCP server | SEC API hosted MCP |
| Need a stable handoff from search to retrieval | SEC API filing search, then accession-based retrieval |
| Need the original filing as the source of record | Keep the SEC filing URL and accession number regardless of the interface |

Do not select an interface from a benchmark number alone. Compare the exact issuer, filing family, section, response fields, and source record your production workflow needs.

## Prove one filing workflow

Use a known issuer and form. SEC API's latest-filing response is time-dependent, so verify the returned filing identity rather than expecting a fixed result.

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K&view=agent"
```

For a bounded question, fetch a section from that current filing:

```bash
curl --fail-with-body -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest/sections/item_1a?ticker=AAPL&form=10-K&view=agent"
```

Persist `accessionNumber`, `form`, `filingDate`, `filingUrl`, and `requestId` when returned. The section is evidence for a specific question; it does not replace the filing identity or the original source document.

## What to compare before choosing

| Check | Why it matters |
| --- | --- |
| Issuer identity | Confirm ticker, CIK, company name, and the identifiers your application joins downstream. |
| Filing identity | Compare form, filing date, accession number, and SEC filing URL. |
| Extraction need | Test the exact section or structured value your workflow consumes, including missing-data behavior. |
| Runtime ownership | Decide whether your team operates local Python parsing, MCP, and caches or consumes a hosted managed route contract. |
| Evidence record | Retain provenance, warnings, freshness, and request diagnostics when the chosen route returns them. |

SEC API does not promise that every exhibit, historic filing representation, or normalized field is available for every issuer. Confirm route-specific availability from the response, and keep a direct-source fallback when your acceptance test requires one.

## Choose the operating boundary deliberately

edgartools can be the better fit when a team wants to own Python execution, local storage, direct SEC retrieval, and its local MCP/LLM workflow. SEC API can be the better fit when a team wants a managed HTTP or hosted MCP surface from JavaScript, Python, Go, Rust, a CLI, or another HTTP-capable runtime.

Local caching and hosted retrieval answer different operational questions. Compare cache refresh policy, source identity, error handling, and the exact filing representation your users need. During a migration, run one workflow side by side and compare evidence records before changing downstream storage or prompts.
