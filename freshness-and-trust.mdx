# Freshness and trust

<Info>
Trust comes from source-backed answers, explicit freshness, stable contracts, and enough metadata to keep the trail intact after the data leaves the API.
</Info>

The current trust evidence supports scoped claims around benchmark performance, provenance-first contracts, filing-page traceability, and a 9-year searchable filing and section floor. It does not support minute-level freshness, universal real-time coverage, or a claim that years 10-20 carry the same hot-path density guarantee. Freshness is measured, not implied.

## Measurement policy

<Steps>
  <Step title="Source publish time">
    Use the upstream SEC publish timestamp, such as `acceptanceDateTime`, as the source-side clock.
  </Step>
  <Step title="API availability">
    Measure the first successful SEC API resolution of the workflow being benchmarked.
  </Step>
  <Step title="Normalized availability">
    Track when the rendered or normalized artifact is ready, not just when a raw fetch succeeds.
  </Step>
</Steps>

## What makes a response trustworthy

<CardGroup cols={2}>
  <Card title="Provenance" icon="badge-info">
    If SEC API gives you a structured answer, it should also tell you which filing or source object it came from.
  </Card>
  <Card title="Freshness" icon="clock-3">
    If SEC API gives you a current answer, it should also tell you when the source was published and when SEC API last refreshed it.
  </Card>
  <Card title="Materialization" icon="layers-2">
    If a parser or normalization pass shaped the output, the version should be visible.
  </Card>
  <Card title="Failure states" icon="shield-alert">
    If a workflow is stale or degraded, the response should say so instead of pretending nothing is wrong.
  </Card>
</CardGroup>

## XBRL facts architecture

Financial data queries (facts and statements) use a DB-first architecture. Over 95 million pre-parsed XBRL facts across 9,600+ SEC-registered companies are stored in SEC API's database and refreshed daily at 07:00 UTC.

The query path:
1. Check if the company has data in the `xbrl_facts` table
2. If data exists, query directly from the database (fast path)
3. If no data exists, fall back to live SEC EDGAR fetch (slow path)

Every response includes a `source` field in its provenance or completeness object: `"xbrl_facts"` for database-served responses, `"company_facts"` for SEC-served responses.

## Evidence used for claims

SEC API trust claims are based on freshness telemetry, external-reference validation, benchmark scorecards, and coverage evidence. Treat claims without matching evidence as unsupported.
