# Freshness and trust

<Info>
Trust is not a slogan here. It is the combination of source-backed answers, explicit freshness, stable contracts, and enough metadata to keep the trail intact after the data leaves the API.
</Info>

The current checked-in trust artifacts support strong scoped claims around benchmark performance, provenance-first contracts, filing-page traceability, and a 9-year searchable filing/section floor. They do not support minute-level freshness, universal real-time coverage, or a claim that years 10-20 carry the same hot-path density guarantee. Freshness is still an explicitly measured posture, not a hand-waved marketing claim.

Factor workflows have additional market-calendar and model-materialization semantics. See [Factor Provenance](/factor-provenance) and [Factor Freshness](/factor-freshness) for the factor-specific trust contract.

## Measurement policy

<Steps>
  <Step title="Source publish time">
    Use the upstream SEC publish timestamp, such as `acceptanceDateTime`, as the source-side clock.
  </Step>
  <Step title="API availability">
    Measure the first successful SEC API resolution of the workflow being benchmarked.
  </Step>
  <Step title="Normalized availability">
    Track when the rendered or normalized artifact is ready, not just when a raw fetch succeeds.
  </Step>
</Steps>

## Freshness classes

Freshness claims are scoped to a specific customer-visible class. A filing can be discovered before its rendered text is ready, and a rendered filing can be searchable before its XBRL statements or semantic vectors are refreshed.

| Class | What it means | Primary evidence |
| --- | --- | --- |
| Raw SEC discovery | SEC API has observed the filing or source record from SEC feeds or SEC endpoints. | source-ingest records, filing provenance `acceptedAt`, hot-lane/RSS telemetry |
| Metadata availability | The filing appears in SEC API filing search/latest responses with CIK, ticker, form, date, accession, and source URL. | the filing manifest store, `/v1/filings`, `/v1/filings/latest` |
| Filing HTML/text availability | The filing body can be rendered or read from persisted storage without waiting on a live SEC fetch. | rendered filing routes, manifest render-depth, storage/read smoke |
| Section extraction availability | Standard 10-K/10-Q/8-K/DEF 14A item sections are available with item labels and citations. | the section-snippet store, section manifests, section routes |
| XBRL/statement availability | Facts and normalized statements are available from the DB-first XBRL store or, when missing, from the explicit SEC companyfacts fallback. | the XBRL facts store, statement/facts provenance |
| Semantic vector availability | Section content has been embedded with finance-tuned embeddings and is present in the vector-search section namespace for semantic/hybrid search. | vector index stats, vector spot checks, bounded reindex artifact |

## Claim mapping

| Public wording | Freshness class | Launch posture |
| --- | --- | --- |
| "Latest filings" and filing search | Metadata availability | Supported for core forms; measured by freshness telemetry and live filing hot-path benchmarks. |
| "Filing text" or "rendered filing" | Filing HTML/text availability | Supported when storage-backed or successfully rendered; stale/missing derived text should surface as degraded rather than blocking unrelated metadata reads. |
| "Section/item extraction" | Section extraction availability | Supported for standard item keys on core forms; freshness is separate from raw filing discovery. |
| "Normalized statements" and "XBRL-to-JSON" | XBRL/statement availability | Supported through DB-first XBRL facts with explicit fallback/source metadata. |
| "Semantic search" and "AI retrieval" | Semantic vector availability | Supported when vector providers are configured and the vector index is current; keyword fallback must disclose `partial_vector_search`. |
| "Real-time" or "webhook freshness" | Raw SEC discovery plus delivery evidence | Do not generalize this claim across all classes without delivery retry evidence and measured lag for the specific event stream. |

## Freshness status semantics

SEC API freshness metadata uses four status values:

| Status | Meaning |
| --- | --- |
| `fresh` | The source-to-visible lag is inside the class threshold for the workflow. Many filing-derived paths currently use a 24-hour fresh threshold unless a stricter class-specific SLO is published. |
| `degraded` | The artifact is usable but outside the fresh threshold on routes that expose a degraded band. Not every runtime path currently emits this intermediate state. |
| `stale` | The artifact is outside the freshness threshold for the route and should not be treated as current. Some filing metadata paths move directly from `fresh` to `stale`. |
| `unknown` | SEC API does not yet have enough source-published or visibility timing to compute the class. First-response paths can return `unknown` while asynchronous visibility telemetry is being recorded. |

Default/full SEC data responses preserve provenance, freshness, and materialization metadata where the route has those fields. Compact and `view=agent` responses may omit some envelope metadata by design to reduce token cost; they keep citation fields needed for downstream auditability.

## What makes a response trustworthy

  <Card title="Provenance" icon="badge-info">

  </Card>
  <Card title="Freshness" icon="clock-3">
    If SEC API gives you a current answer, it should also tell you when the source was published and when SEC API last refreshed it.
  </Card>
  <Card title="Materialization" icon="layers-2">

  </Card>

## XBRL facts architecture

Financial data queries (facts and statements) use a DB-first architecture. Over 95 million pre-parsed XBRL facts across 9,600+ SEC-registered companies are stored in our own PostgreSQL database and refreshed daily via our scheduled data pipeline at 07:00 UTC.

The query path:
1. Check if the company has data in the XBRL facts store
2. If data exists, query directly from the database (fast path)
3. If no data exists, fall back to live SEC EDGAR fetch (slow path)

Every response includes a `source` field in its provenance or completeness object: `"xbrl_facts"` for database-served responses, `"company_facts"` for SEC-served responses.
