Use this page to decide what a response can support. SEC API separates three questions that are often blurred together: when the SEC published a source, when SEC API could return it, and when a derived representation such as a section, statement, or semantic-search result became available.

Do not treat an available filing manifest as proof that every derived surface is current. Keep the filing accession, source URL, request ID, and any returned freshness or degradation metadata with a downstream conclusion.

## Read freshness by data class

| Data class | What availability means | What to preserve |
| --- | --- | --- |
| Filing metadata | The filing record can be found by issuer, form, date, or accession. | `accessionNumber`, `filingDate`, `filingUrl`, `requestId` |
| Filing text | A filing body can be rendered or read from the available source/storage path. | accession, source URL, render state when present |
| Extracted sections | A supported filing section can be returned with its filing identity and citation fields. | accession, section key, source URL, citation span when present |
| Facts and statements | A normalized value is available from the documented database path or SEC fallback. | provenance, period, accession/source fields, materialization metadata when present |
| Semantic search | A section is available to the configured retrieval path. | query, mode, accession, source URL, citation fields, degraded state |

## Freshness is measured, not assumed

The latest checked-in freshness dashboard was generated on **2026-03-24** and records a `ready` dashboard state for its measured forms and workflows. It is evidence about that dated run, not a standing promise of minute-level or universal real-time availability. Re-run or refresh the telemetry before publishing a newer lag claim.

The public history contract is separate from freshness: it sets a 9-year searchable filing and section floor, while only the two most recent completed years are release-blocking completeness guarantees. Inventory coverage beyond that floor is not an equal-density production guarantee. See [Coverage and Historical Depth](/coverage-and-depth).

## Response states

Where a route exposes freshness status, the documented values are:

| Status | What it means for the route |
| --- | --- |
| `fresh` | The route considers the artifact inside its documented freshness threshold. |
| `degraded` | The response is usable, but a route-specific freshness or dependency condition is outside its normal posture. |
| `stale` | The artifact should not be treated as current for that route. |
| `unknown` | The route does not have enough timing information to classify the artifact. |

Not every endpoint has this field, and compact or `view=agent` projections can omit envelope metadata. Use the endpoint reference for the exact response shape; do not infer freshness from the absence of an error.

## What SEC API can prove

- **Source lineage:** filing, accession, and source URLs are available on documented filing and search paths.
- **Derived-data posture:** routes can expose provenance, materialization, or degradation metadata where the contract includes it.
- **Request correlation:** SEC API app responses include `Request-Id` and `traceparent`; supplied correlation IDs are for tracing, not billing idempotency.
- **Coverage boundary:** source-backed filing workflows and derived surfaces have different availability conditions. A section, XBRL fact, or vector result should be evaluated in its own class.

## XBRL facts and statements

Facts and statement workflows use a database-first SEC data path with a documented SEC company-facts fallback when data is unavailable in the primary store. We publish exact corpus counts only when they are tied to a dated retained count artifact; do not infer a current count or full-universe completeness from this architecture.

## Use this in production

1. Resolve the issuer and retrieve the filing or data point you need.
2. Store the identifiers and source fields that arrive with that response.
3. Check route-specific freshness, provenance, and degradation fields before automating a decision.
4. On an unexpected result, preserve `Request-Id` and use [Request Diagnostics](/request-diagnostics) before opening support.
