title: "Freshness and source evidence"
description: "Use route-specific timing, provenance, and degradation fields to decide whether an SEC API result meets your workflow's evidence policy"
---

An SEC filing being published, an API record being available, and a derived representation being ready are different events. For a material decision, retrieve the specific result you need and retain the source and state fields that describe it.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"
```

## Build a result record

When a route returns them, retain the filing accession number, filing date, and filing URL. Record the endpoint, parameters, response view, period, units, provenance, capability, warning, freshness, and degradation fields alongside the result. Keep the `Request-Id`, `traceparent` when present, pinned API version when used, and retrieval time in UTC.

Field names and availability vary by endpoint and response view. The API reference defines the contract for a route. A filing manifest by itself does not establish that its sections, statements, search results, or traces are available.

## Read freshness in the context of its route

Use a freshness status only when that route returns one. Depending on the route, it can describe a source, a materialized artifact, or another route-specific condition. It is not a platform-wide latency commitment.

| Status | Practical interpretation |
| --- | --- |
| `fresh` | Current under the route's documented policy. |
| `current` | A recently published immutable filing was materialized without a lag indication. |
| `archival` | A historical immutable filing is available; its age alone does not make it stale. |
| `stale` | Outside the route's stated freshness posture. |
| `degraded` | Returned with a freshness or dependency limitation. |
| `unknown` | The route cannot classify the condition from available timing information. |

An endpoint can support only some of these values. Do not infer a status, real-time behavior, or a minute-level target for one route from another route's fields.

## Apply an evidence policy

1. Define the source event and timing rule your workflow requires.
2. Retrieve the specific filing, section, fact, or derived output for that rule.
3. Compare the returned source dates, periods, and route-specific state with your rule.
4. Store the request and source identifiers with the decision or downstream model output.
5. Route missing, stale, degraded, or unknown evidence to review instead of displaying it as current.

Facts and statements normalize issuer disclosures, but what is available depends on the disclosure and the route's materialization path. Check reporting period, units, `periodBasis` when present, and provenance before comparing values.
