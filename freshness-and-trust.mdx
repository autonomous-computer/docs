---
title: "Freshness and source evidence"
description: "Evaluate an SEC API result using its route-specific source, timing, and degradation evidence"
---

A filing can be published before it appears in a route, and a route can return a filing before a derived representation is ready. Treat those as separate facts. The API reference is the contract for a route; this page explains how to decide whether the returned evidence meets your own decision policy.

```bash
curl -sS -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"
```

## Record the result, not an assumption

For a material workflow, retain the endpoint, parameters, retrieval time in UTC, response view, source identifiers, period, units, and any provenance, capability, warning, freshness, or degradation fields returned. Keep `Request-Id`, `traceparent` when present, and a pinned API version when your integration uses one. Store the response or a stable extract of those fields when your retention policy permits, so the decision can be reviewed without reconstructing it from memory.

Field names and availability vary by route and response mode. Do not manufacture absent fields or infer a platform-wide clock from a status on another route.

## Interpret timing in context

| Returned state | Practical treatment |
| --- | --- |
| `fresh` or `current` | Usable only under that route's documented timing policy. |
| `archival` | Historical source material can remain valid even though it is old. |
| `stale` | Outside the route's stated freshness posture; do not label it current. |
| `degraded` | A response with a stated limitation, fallback, or dependency issue. |
| `unknown` | The route cannot classify the condition from the information available. |

An endpoint may expose only a subset of these states. A healthy HTTP response is not proof of source recency, complete coverage, or analytical fitness.

## Apply an evidence policy

1. Define the source event, period, and maximum acceptable lag before retrieval. For example, distinguish a decision that needs the latest filed quarterly result from one that needs a particular historical filing.
2. Retrieve the filing, section, fact, or derived output that answers that question; avoid using an adjacent route as a proxy for the requested representation.
3. Compare returned source dates, periods, units, and route-specific state with the policy. Record both the source date and retrieval time because they answer different questions.
4. Store the request and resulting identifiers with the downstream decision, report, or model output.
5. Escalate missing, stale, degraded, or unknown evidence rather than presenting it as current. A retry can resolve a transport failure, but it does not convert an unavailable source or incomplete representation into evidence.

Normalized facts and statements still reflect issuer disclosures. Compare reporting period, units, `periodBasis`, and provenance when supplied; do not combine values until their time bases and units are compatible. A filing manifest alone does not prove that its sections, statements, search results, or traces are available.
