---
title: "Freshness and source evidence"
description: "Evaluate route-specific source, timing, provenance, and degradation information before relying on an SEC API result"
---

Start with the source record for the result you need, then preserve the fields that describe how that result was produced. SEC filing publication, API availability, and availability of a derived representation are separate facts.

```bash
curl -sS \
  -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/filings/latest?ticker=AAPL&form=10-K"
```

## Build an audit record from the response

Keep the source filing's accession number, filing date, and filing URL when returned. Alongside them, record the endpoint, parameters, response view, period, units, and route-specific provenance, capability, warning, freshness, or degradation fields. Keep `Request-Id`, `traceparent` when present, the API version when pinned, and the retrieval time in UTC.

Field names and field presence vary by endpoint and view. The endpoint reference is the contract. A filing manifest alone does not establish that every section, statement, search result, or trace is available.

## Treat freshness as route-specific

Only use a freshness status where the route returns one. Depending on the route, status can describe a source, materialized artifact, or other route-specific condition. It is not a platform-wide promise about every SEC API surface.

Where a route documents these statuses, use the returned reason fields with the status:

| Status | Meaning |
| --- | --- |
| `fresh` | Current under that route's own policy. |
| `current` | A recently published immutable filing was materialized without a lag indication. |
| `archival` | A historical immutable filing is available; age alone is not stale. |
| `stale` | Outside the route's stated freshness posture. |
| `degraded` | Available with a freshness or dependency limitation. |
| `unknown` | The route cannot classify the condition from available timing information. |

An endpoint can expose only a subset of these values. Do not infer support for a status, or a universal real-time or minute-level commitment, from another route.

## Decide whether the evidence meets your policy

1. Define the event and timing rule your workflow requires.
2. Retrieve the specific filing, section, fact, or derived output needed for that rule.
3. Compare returned source dates, periods, and route-specific state with your policy.
4. Record the request and source identifiers with the decision or downstream model output.
5. Send missing, stale, degraded, or unknown evidence to a review path rather than presenting it as current.

Facts and statements are normalized from issuer disclosures, but availability depends on what was disclosed and the route's materialization path. Check reporting period, units, `periodBasis` where available, and provenance before comparing values.
