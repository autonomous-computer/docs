Return structured Form D offering data parsed from the filing XML

<Info>
Audience: application and coding agent.
</Info>

## Coverage and source scope

Use this route after selecting a Form D accession to inspect structured values parsed from that filing's XML. The result reflects the filed notice, not independent verification that a financing closed, that all securities were sold, or that undisclosed terms do not exist. Retain the accession number and filing source when reporting values, and review the linked filing when a field affects a capital-raise conclusion. See [Form D API guidance](/seo/sec-form-d-api) and [API conventions](/api-conventions).

## Canonical metadata

- `requestId`
- `traceparent`

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/forms/d/0000320193-25-000079"
```
</RequestExample>

## Example response

<ResponseExample>
```json
{
  "requestId": "req_2ZK8Q1W9F4M6P7R3"
}
```
</ResponseExample>

## Give this prompt to your agent

<Prompt>
Use SEC API GET /v1/forms/d/{accessionNumber} after selecting a Form D accession to inspect values parsed from its XML. Preserve the accession number, filing source, `requestId`, and `traceparent`; when present, preserve `provenance`, `freshness`, and `materialization` metadata unchanged, but do not assume those fields on response shapes that omit them. Report values as filed disclosures rather than independent proof of capital-raise completion. Review the cited filing when a parsed field affects a transaction conclusion.
</Prompt>

## Failure posture

- treat non-2xx responses as contract-aware failures, not free-form errors
- preserve `requestId` and `traceparent` in logs and downstream reports
- if provenance or freshness metadata is present, return it unchanged so trust is not lost in the handoff
