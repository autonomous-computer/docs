List saved-search monitors for the current organization

<Info>
Audience: application and coding agent.
</Info>

## Reading monitor state

The list is scoped to the authenticated organization. Each entry carries its saved query, filters, delivery configuration, active state, and `lastCheckedAt`; use those fields to distinguish a configured monitor from evidence of a completed delivery.

## Canonical metadata

- `requestId`
- `traceparent`

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/monitors"
```
</RequestExample>

## Example response

<ResponseExample>
```json
{
  "requestId": "req_2ZK8Q1W9F4M6P7R3"
}
```
</ResponseExample>

## Give this prompt to your agent

<Prompt>
Call SEC API GET /v1/monitors to inspect saved searches in the authenticated organization. Use the returned ids, active state, search mode, delivery status, and lastCheckedAt to decide which monitor to read or change next.
</Prompt>

## Failure posture

- treat non-2xx responses as contract-aware failures, not free-form errors
- preserve `requestId` and `traceparent` in logs and downstream reports
- if provenance or freshness metadata is present, return it unchanged so trust is not lost in the handoff
