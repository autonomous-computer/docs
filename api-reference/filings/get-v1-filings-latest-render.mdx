# GET /v1/filings/latest/render

Render the latest filing into Markdown-like text

<Info>
Audience: application and coding agent.
</Info>

## Issuer lookup

Use `ticker`, `symbol`, or `cik` to choose the issuer. `symbol` is an alias for `ticker`; if both are provided, they must match.

## Response size

Rendered filings can exceed typical agent context budgets. Prefer section extraction with `view=agent` when you only need a specific item, and use this full render endpoint when the downstream workflow truly needs the whole filing body.

## Canonical metadata

- `requestId`
- `traceparent`
- `provenance`
- `freshness`
- `materialization`

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/filings/latest/render?symbol=AAPL&form=10-K&filing_year=2025"
```
</RequestExample>

## Give this prompt to your agent

<Prompt>
Use SEC API GET /v1/filings/latest/render with `ticker`, `symbol`, or `cik` to render the latest filing into Markdown-like text. Preserve `requestId`, `traceparent`, `provenance`, `freshness`, `materialization` in the output. Return a concise markdown summary plus the exact structured payload fields that a downstream engineer or agent should keep using this result.
</Prompt>

## Failure posture

- treat non-2xx responses as contract-aware failures, not free-form errors
- preserve `requestId` and `traceparent` in logs and downstream reports
- if provenance or freshness metadata is present, return it unchanged so trust is not lost in the handoff
