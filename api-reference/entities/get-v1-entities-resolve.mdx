# GET /v1/entities/resolve

Resolve an entity by ticker, CIK, FIGI-family identifier, ISIN, CUSIP, or name with confidence and match-basis metadata.

<Info>
Use this endpoint from server-side applications, SDKs, or agent workflows. Provide at least one identifier: `ticker`, `cik`, `figi`, `composite_figi`, `share_class_figi`, `isin`, `cusip`, or `name`.
</Info>

## Agent mode

Pass `view=agent` to receive the compact agent shape: entity identifiers (`id`, `ticker`, `cik`, `name`), primary identifiers, and match metadata. Aliases, jurisdiction, full identifiers list, and metadata envelopes are dropped. See [api conventions → response formats](/api-conventions) for the full `view` contract.

## Response metadata

- `requestId`
- `traceparent`
- `provenance`
- `freshness`
- `materialization`

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL&view=agent"
```
</RequestExample>

<RequestExample>

## Give this prompt to your agent

Use SEC API GET /v1/entities/resolve with `ticker=AAPL` or the identifier the user provides. Pass `view=agent` unless the workflow needs the full entity record. Preserve `requestId`. Return the resolved `id`, `ticker`, `cik`, `name`, `matchConfidence`, `matchBasis`, and primary identifiers.

## Errors and partial results

- Treat non-2xx responses as API errors and log the response body for debugging
- If the response is `missing_identifier`, retry with one of `ticker`, `cik`, `figi`, `composite_figi`, `share_class_figi`, `isin`, `cusip`, or `name`
- Log `requestId` so SEC API support can trace the request
