# GET /v1/statements/segmented-revenues

Return filing-derived segmented revenue history with XBRL product or geography dimensions, capability state, and trace references when issuers disclose them

<Info>
Use this endpoint from server-side applications, SDKs, or agent workflows.
</Info>

## Response metadata

- `requestId`
- `traceparent`
- `provenance`
- `freshness`
- `materialization`

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/statements/segmented-revenues?ticker=PFE&period=q&segment_type=geographic&limit=3&submission_file_limit=4"
```
</RequestExample>

## Period aliases

Use `period=annual` for annual segments or `period=quarterly` for quarterly segments. The shorter `period=quarter` and `period=q` aliases are also accepted for quarterly data.

## Query parameters

- `ticker` or `cik` identifies the issuer. One is required.
- `segment_type` can be `geographic`, `product`, or `other`.
- `limit`, `segment_limit`, or `segmentLimit` caps segment periods returned, up to `6`.
- `submission_file_limit` caps recent SEC submission files inspected, up to `6`. `submissionFileLimit` is also accepted.

## Give this prompt to your agent

<Prompt>
Use SEC API GET /v1/statements/segmented-revenues with `ticker` or `cik` and optional `segment_type=geographic|product|other` to return bounded filing-derived revenue segment history. Use `period=quarter`, `period=q`, or `period=quarterly` for quarterly segments. Preserve `requestId`, `traceparent`, `provenance`, and record-level `accessionNumber` values.
</Prompt>

## Failure posture

- treat non-2xx responses as contract-aware failures, not free-form errors
- preserve `requestId` and `traceparent` in logs and downstream reports
- invalid `period` and `segment_type` values return `400 invalid_query_parameter` with `details.acceptedValues`
- if provenance or freshness metadata is present, return it unchanged so trust is not lost in the handoff
