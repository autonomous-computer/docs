# GET /v1/signals/dilution

Return the OMNI Dilution Score: factor-weighted 0-100 derived metric with sub-ratings, percentile, and optional rolling history

<Info>
Audience: application and coding agent.
</Info>

## Canonical metadata

- `requestId`
- `traceparent`
- `provenance`
- `freshness`
- `methodologyVersion`
- `factors`
- `subRatings`
- `regsho`
- `percentile`
- `degradedReasons`

## Query parameters

| Param | Type | Description |
|---|---|---|
| `ticker` | string | Issuer ticker. Either `ticker` or `cik` is required. |
| `cik` | string | Issuer CIK (10-digit zero-padded). Either `ticker` or `cik` is required. |
| `history` | enum | Optional. `30d` \| `90d` \| `1y`. Returns rolling timeseries under `history[]`. |
| `view` | enum | Response shape (`default` \| `compact` \| `agent`). `agent` drops `factors[]` for token efficiency. |

## Example request

<RequestExample>
```bash
curl -X GET \
  -H "x-api-key: $OMNI_DATASTREAM_API_KEY" \
  -H "omni-version: 2026-03-19" \
  "https://api.secapi.ai/v1/signals/dilution?ticker=MULN"
```
</RequestExample>

## Give this prompt to your agent

<Prompt>
Use Omni Datastream GET /v1/signals/dilution to return the OMNI Dilution Score for an issuer. The response is a single 0-100 derived metric with a band (low/moderate/elevated/high), confidence rating, factor breakdown, six categorical sub-ratings (overallRisk, offeringAbilityRisk, dilutionRisk, historicalRisk, cashNeedLevel, warrantExerciseRating), regsho boolean, percentile rank, and provenance. Preserve `requestId`, `traceparent`, `provenance`, `freshness`, `methodologyVersion`, `factors`, `subRatings`, `regsho`, `percentile`, `degradedReasons` in the output. Use `?history=30d|90d|1y` for rolling timeseries reads.
</Prompt>

## Failure posture

- treat non-2xx responses as contract-aware failures, not free-form errors
- preserve `requestId` and `traceparent` in logs and downstream reports
- when `confidence` is `low` or `freshness.status` is `stale`/`degraded`, surface the `degradedReasons[]` to the user verbatim so trust is not lost in the handoff
- the score is deterministic given (inputs, methodologyVersion); a methodology bump is recorded in the persisted history table without overwriting prior rows
