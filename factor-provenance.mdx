---
title: "Factor provenance"
description: "Preserve the source, methodology, materialization, and degraded-state metadata needed to interpret SEC API factor results"
---

Factor results can be small enough for a chart or agent response while still needing an audit trail. When the workflow needs source and methodology context, request `include=trust` with a compact response.

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19" \
  "https://api.secapi.ai/v1/factors/returns?keys=VALUE,MOMENTUM,QUALITY&lookback=6m&response_mode=compact&include=trust"
```

For MCP, carry the same intent in the tool arguments:

```json
{
  "name": "factors.returns",
  "arguments": {
    "keys": ["VALUE", "MOMENTUM", "QUALITY"],
    "lookback": "6m",
    "response_mode": "compact",
    "include": ["trust"]
  }
}
```

## Preserve these fields

| Field | Why to keep it |
| --- | --- |
| `requestId` | Identifies the API request when an outcome needs investigation. |
| `traceparent` | Connects the API request to downstream tracing when your system uses it. |
| `freshness` | States whether the result is fresh, stale, degraded, or unknown for its applicable clock. |
| `materialization` | Identifies the parser, model, or read-model version that produced the result. |
| `methodology` | Identifies the model family, inputs, and construction notes when included. |
| `sourceRights` | Describes the source-rights posture returned with the result. |
| `degradedState` | Explains a partial response, fallback, or unavailable dependency. |

## Read the source posture by factor family

| Factor family | Inputs used by the documented model |
| --- | --- |
| Market | SEC API market-data inputs and benchmark construction metadata |
| Style | Point-in-time security returns and public-company characteristics, including SEC-derived fundamentals when the definition uses them |
| Sector and industry | Point-in-time security returns and public classification mappings, including SIC-derived groupings |
| Portfolio attribution and hedge | Caller-supplied holdings plus SEC API factor exposure and return read models |
| Thematic, country, and macro | Catalog or methodology definitions; not every definition is part of the launch-read default surface or public history claim set |

Do not treat a catalog entry as proof that a factor is available for every surface or period. Confirm row-level materialization, freshness, model/version metadata, and returned source-rights posture for the result you use.

## Respond to materialization changes

The `materialization.materializationVersion` value identifies the read model behind a result. Treat it as opaque. Store it and detect when it changes, because a revision to methodology or source data can change historical numbers.

Current read models cover stock-basket returns, purified returns, latest stock-basket exposures, latest purified exposures, portfolio attribution, and portfolio hedge results. The version distinguishes a materialized read model, a computed fallback, or an explicit degraded path when the response exposes that distinction.

## Do not hide a degraded result

A degraded response is a structured outcome, not a silent success and not necessarily a generic error. Examples include unavailable factor returns when a required read model is not materialized, partial portfolio attribution when holdings lack sufficient market bars, empty hedge suggestions when exposures are missing or constraints reject all candidates, and stale intraday state when market-data inputs are not current.

When present, preserve `degradedState.code`, `degradedState.message`, `degradedState.retryable`, `degradedState.missingCapabilities`, and `degradedState.fallbackUsed`. Some materialization errors can also include `operatorAction`; retain it when the API returns it outside the degraded-state envelope.
