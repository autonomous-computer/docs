---
title: "Record the SEC API version you use"
description: "Send and log the SEC API version header so production behavior can be compared and investigated precisely."
---

`secapi-version` is a request header. The service resolves a version for the request and returns it in `SECAPI-Version`. Use those two values as operational evidence, not as a substitute for testing an endpoint contract.

## Prerequisites

- An API key in `SECAPI_API_KEY`.
- A known working endpoint.

## Send and inspect the version

```bash
curl --include --fail-with-body \
  "https://api.secapi.ai/v1/entities/resolve?ticker=AAPL" \
  -H "x-api-key: $SECAPI_API_KEY" \
  -H "secapi-version: 2026-03-19"
```

Expect a successful response and headers similar to:

```text
Request-Id: req_...
SECAPI-Version: 2026-03-19
```

If you omit the request header, the service uses its current version value and still emits `SECAPI-Version`. Record the emitted value in deploy logs before comparing behavior across environments.

## Use it in a client

```ts
import { SecApiClient } from "@secapi/sdk-js"

const sec = new SecApiClient({
  apiKey: process.env.SECAPI_API_KEY,
  apiVersion: "2026-03-19",
})

const issuer = await sec.resolveEntity({ ticker: "AAPL" })
console.log(issuer.ticker)
```

The JavaScript and Python clients expose an API-version option. Check the client documentation and emitted request headers when upgrading a client package.

## When something changes

1. Reproduce the request with the same method, path, parameters, and response view.
2. Capture `Request-Id`, `SECAPI-Version`, status, and the error `code` when present.
3. Compare the affected operation's [API reference](/api-reference) and the [changelog](/changelog).
4. Test any new client or application version in a controlled environment before moving production traffic.

## Limits and production notes

- A version header does not make an undocumented field or an endpoint-specific projection universal.
- Treat new optional fields and unknown enum values defensively.
- Keep version configuration near your client construction and log the response version with each failed request.
- Do not infer a release policy from an echoed header; use published migration guidance when it exists.
