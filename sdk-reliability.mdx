title: "SDK reliability"
description: "Understand retries, backoff, retry budgets, and circuit breakers in the SEC API JavaScript and Python SDKs"
---

The JavaScript and Python SDKs retry transient read failures so a brief network or service interruption does not immediately fail a filing lookup. `POST`, `PUT`, `PATCH`, and `DELETE` are not retried for transient failures by default. A `429` rate-limit response is the exception: unless retries are disabled, the SDK can reissue any method after the retry delay.

## Default retry policy

| Behavior                      | Default                                                            |
| ----------------------------- | ------------------------------------------------------------------ |
| Automatically retried methods | `GET`, `HEAD`, `OPTIONS`                                           |
| Retryable failures            | Network failures and HTTP `408`, `429`, `502`, `503`, `504`        |
| Never retried                 | HTTP `400`, `401`, `403`, `404`, `422`                             |
| `429` rate limit response     | Retried for any method unless retries are disabled                 |
| Retry count                   | Up to 3 retries                                                    |
| Backoff                       | Jittered exponential delay, starting at `200ms` and capped at `5s` |
| Retry budget                  | `30s`                                                              |
| Circuit breaker               | Opens after 5 terminal retryable failures and cools down for `60s` |

When the service sends `Retry-After`, the SDK uses it. It also reads structured retry timing from the error payload when an intermediary has removed that header.

## Keep writes explicit

`POST`, `PUT`, `PATCH`, and `DELETE` are not retried for transient failures by default. A `429` can still reissue any method unless you disable retries. Disable retries for an operation when replay would be unsafe or when another layer owns retry behavior.

Enable a retry for a write only when your application can safely repeat it. Provide an idempotency key when the endpoint supports one, and keep the key stable for the logical operation.

```ts
const client = new SecApiClient({
  apiKey: process.env.SECAPI_API_KEY,
  retry: false,
});

await client.callMcpTool(
  "entities.resolve",
  { ticker: "AAPL" },
  { retry: { enabled: true } },
);
```

MCP tool calls use `POST`. Opt in to retries only for a read-only tool or an operation that your application has made idempotent.

## Configure retry ownership

Disable SDK retries when another layer already owns retries and backoff. This avoids multiplying attempts during an outage.

<CodeGroup>
```ts TypeScript
const client = new SecApiClient({
  apiKey: process.env.SECAPI_API_KEY,
  retry: false,
})
```

```python Python
import os

from secapi_client import SecApiClient

client = SecApiClient(
    api_key=os.environ["SECAPI_API_KEY"],
    retry=False,
    timeout=10,
)
```

</CodeGroup>

The Python SDK uses a `30s` per-request socket timeout by default. Set a shorter timeout for latency-sensitive work, or leave it unset only when your transport layer enforces a deadline. The JavaScript SDK applies its retry budget to the request attempt sequence.

## Inspect a failure

Both SDKs surface the HTTP status, API error code, request ID, and response payload on API failures. Log the request ID with the operation that failed. When the client circuit is open, wait for its cooldown rather than creating parallel client instances to bypass it.

The JavaScript SDK emits a `client_retry_attempt` event only when a telemetry capture token is configured. The Python SDK emits retry telemetry by default; set `telemetry=False` or `telemetry={"enabled": False}` to disable it.
