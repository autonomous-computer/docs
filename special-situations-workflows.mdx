Use Special Situations as a durable SEC-derived research layer: discover a live population, open the cited event trail behind a candidate, and preserve a fixed weekly archive when reproducibility matters.

<Info>
This is a paid public API product. The free [Situations database](https://secapi.ai/situations) and anonymous Embed routes are useful for logged-out discovery; an API key unlocks programmatic research, exports, archive reads, and underwriting under your organization plan.
</Info>

## Choose the right object

| Need | Start with | Why |
| --- | --- | --- |
| Scan live candidates | `GET /v1/situations` | A current, filterable roster of durable situations |
| Explain a current candidate | `GET /v1/situations/{situation_id}` | Current terms, narrative, market context, and filing timeline |
| Cite a concise agent brief | `GET /v1/situations/{situation_id}/export` | Source-cited Markdown designed for downstream models |
| Build a single research packet | `GET /v1/situations/{situation_id}/underwriting-pack` | Canonical detail, narrative, timeline, and Copy-for-LLM Markdown together |
| Reproduce a past digest | `GET /v1/situations/issues/{issue}` | Immutable publication-time snapshot, not live data |

## Keep the public boundary explicit

| Surface | Available work | Not a substitute for |
| --- | --- | --- |
| Anonymous Embed | Recent, redacted list, detail, feed, stats, issue archive, and Markdown brief | The authenticated roster, full historical issue archive, or underwriting |
| Authenticated REST | Published list/detail/feed/issues/calendar/stats/export/underwriting operations | A claim about transaction completion or investment suitability |
| CLI and JavaScript SDK | Published shell and client helpers for the authenticated operations | A browser-delivered credential or a public embed |
| Hosted MCP | Discover tools from the server before invoking them | A substitute for the published REST reference |

Anonymous Embed reads use no API key and are intentionally capped, recent-window, and redacted. Treat `404` as unavailable rather than falling back to an authenticated result or inventing missing terms.

## 1. Discover, then narrow

Use the lean projection for a broad roster when a smaller response is sufficient.

```bash
secapi situations list \
  --types merger,tender_offer \
  --statuses announced,pending \
  --enrich false \
  --limit 100
```

When the investigation begins with a filing type rather than a company, let the server expand the EDGAR form to the situations it can trigger:

```bash
secapi situations by-form --form "SC 13D" --limit 25
secapi situations calendar --date-types vote,expected_close --days 30
```

## 2. Build a source-cited research packet

Retrieve current detail before asking an agent to interpret a situation. Keep the `situationId`, accession numbers, source URLs, event dates, `requestId`, and `traceparent` with the resulting memo. An export is a factual brief, not investment advice and not a substitute for source-filing review.

```bash
secapi situations get --situation-id sit_abc123
curl --fail-with-body -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/situations/sit_abc123/export" > situation-brief.md
```

<Prompt>
Read the cited SEC API Special Situations payload and its Copy-for-LLM export. Separate filing-supported facts from inference. Preserve every accession number, source URL, event date, requestId, and traceparent. State which facts are current live detail versus historical archive data. Do not invent consideration, timing, premium, counterparty, or completion facts when the field is absent.
</Prompt>

## 3. Use immutable issues for reproducibility

Issue data answers a historical question: what did the digest publish then? Live detail answers a current question: what does the situation look like now? Keep those two facts separate in stored research.

```bash
curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/situations/issues?limit=12"

curl -H "x-api-key: $SECAPI_API_KEY" \
  "https://api.secapi.ai/v1/situations/issues/42"
```

The issue number or slug, publication timestamp, `sourceSituationIds`, `sourceEventIds`, and source-event snapshots are the reproducibility anchors. Fetch the referenced `situationId` separately before describing live terms or current status.

## CLI, SDK, MCP, and agent workflows

The published CLI commands are `situations list`, `situations by-form`, `situations feed`, `situations calendar`, `situations stats`, `situations performance`, `situations get`, `situations filings`, and `situations summary`. The published JavaScript `situations` namespace helpers are `list`, `byForm`, `get`, `filings`, and `summary`. Use the [REST reference](/api-reference/situations) for other published Special Situations operations.

For MCP, start with the server's deployed discovery and tool listing rather than a copied tool inventory. Use the published REST reference as the source of truth for routes and response behavior.

For an agent run, use this order: read `GET /v1/limits` and `GET /v1/billing/rates`; discover with list, feed, calendar, or issues; retrieve detail and filings; then request an export or underwriting pack only for selected records. Persist the situation ID, issue number or slug, accession numbers, source URLs, event dates, `requestId`, and `traceparent`. Do not let an agent infer missing deal terms, lifecycle state, or an entitlement from a public embed response.

## Cost and entitlement posture

Before a batch or an agent loop, retrieve `GET /v1/limits` and `GET /v1/billing/rates`, then quote the exact route with `POST /v1/billing/quote` when cadence matters. Retain the returned plan, limits, quote, and request identifiers rather than hard-coding a plan assumption.
